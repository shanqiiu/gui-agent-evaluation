# GUI Agent 规划失效异常判定技术方案

> 版本：v0.2 · 2026-07-15
> 定位：专题设计参考；规范性架构、数据契约和最终输出以 `01-技术方案.md` 为准。
> 目标：针对 GUI Agent 的遗漏关键步骤、误解任务目标、错误子目标顺序、提前终止和未能终止等规划类异常，建立可落地的自动判定方案。  
> 输入边界：优先复用自然语言任务、`step_level_instruction`、实际截图序列、实际动作序列、达尔文 E2E 意图判定和单步判定结果；不要求预置标准轨迹或预期坐标。

规则定位：必要检查点缺失、显式终止时机等可以形成强证据；目标误解、步骤顺序和级联因果必须结合页面状态与语义验证，字符串匹配只用于候选召回。

---

## 1. 问题定义

规划失效是指 Agent 的单步动作可能看起来合理，但完整任务层面的目标分解、步骤选择、执行顺序或终止判断出现错误，导致任务没有按用户意图完成。

本方案优先覆盖五类规划失效：

| 子类型 | 典型表现 | 可观测信号 |
|--------|----------|------------|
| 遗漏必要步骤 | 未保存、未提交、未选择规格、未确认弹窗 | 核心检查点未达成 |
| 误解任务目标 | 搜索错对象、进入无关功能、围绕错误商品执行 | 实际轨迹与任务目标语义偏离 |
| 步骤顺序错误 | 先提交后编辑、未登录直接下单、未选商品进入结算 | 状态转移不满足前置条件 |
| 提前终止 | 部分完成即 `finished` / `done` | 最终页不满足完成条件 |
| 未能终止 | 目标已达成后继续操作 | 完成检查点已达成但仍执行无关动作 |

核心判定原则：

```
任务意图 -> 子目标/检查点 -> 实际页面状态进展

如果核心检查点缺失、进展路径偏离、终止时机错误，
则判为规划失效，而不是简单归因于单步执行错误。
```

## 2. 可用输入

### 2.1 任务与计划输入

| 字段 | 用途 |
|------|------|
| `instruction` | 用户原始任务意图 |
| `step_level_instruction` | 计划步骤，若存在则作为显式检查点来源 |
| RAG 应用知识 | App 私有流程、必要前置条件、业务约束 |

### 2.2 实际执行输入

| 字段 | 用途 |
|------|------|
| `seq_info` | 实际截图与动作序列 |
| `planning_output.parsed_action` | 实际动作类型、坐标、文本、方向 |
| `page_description` | 页面状态语义摘要 |
| `ActionDescription` | 操作目标控件语义 |

### 2.3 达尔文判定输入

| 结果 | 用途 |
|------|------|
| `intention.label` | 整体意图是否达成 |
| `intention.wrong_reason` | 整体失败原因 |
| `*_intention_step` | 每个子步骤是否达成 |
| `*_intention_step_identity` | 路径一致性、缺失步骤、疑似功能 bug |
| `ab_pages_result` | 单步页面跳转是否符合预期 |

## 3. 判定架构

```
用户任务 + 可选 step_level_instruction
  │
  ├─ Plan Decomposer
  │    └─ 生成子目标、核心检查点、前置条件、完成条件
  │
实际轨迹
  │
  ├─ State Interpreter
  │    └─ 截图/页面描述 -> 当前语义状态
  │
  ├─ Progress Aligner
  │    └─ 将每一步状态对齐到子目标进度
  │
  ├─ Transition Validator
  │    └─ 检查步骤顺序和前置条件
  │
  ├─ Termination Checker
  │    └─ 判断 finished/done 是否过早或过晚
  │
  └─ Planning Failure Classifier
       └─ 输出规划失效类型、首错步骤、证据链
```

## 4. 子目标与检查点建模

### 4.1 检查点结构

建议将每个子目标表示为：

```json
{
  "id": "step_2",
  "name": "选择目标商品规格",
  "required": true,
  "preconditions": ["已进入商品详情页"],
  "expected_state": "页面显示目标商品，且规格选中用户要求的值",
  "forbidden_state": "选择了非目标商品或错误规格",
  "evidence": ["目标商品名", "规格选中态", "价格或购物车变化"],
  "weight": 0.25
}
```

### 4.2 检查点来源优先级

| 优先级 | 来源 | 说明 |
|--------|------|------|
| P0 | 人工/测试用例显式 `step_level_instruction` | 最可信 |
| P1 | App RAG 知识库 | 补充业务流程和前置条件 |
| P2 | LLM 从 `instruction` 自动分解 | 用于无显式步骤场景 |
| P3 | 达尔文 `intention_sequence` 自动解析结果 | 与现有框架兼容 |

### 4.3 检查点状态

每个检查点在每个时间步的状态为：

| 状态 | 含义 |
|------|------|
| `not_started` | 尚未出现相关页面或动作 |
| `in_progress` | 进入相关页面，但证据不足 |
| `satisfied` | 检查点已达成 |
| `violated` | 明确进入冲突状态 |
| `unknown` | VLM/规则无法判断 |

## 5. 核心判定算法

### 5.1 进度对齐

对每个时间步 `t`，构建进度向量：

```
P_t = [
  status(checkpoint_1, page_t, action_t),
  status(checkpoint_2, page_t, action_t),
  ...
]
```

进度分数：

```
progress_score_t =
    sum(weight_i for checkpoint_i in satisfied at t)
    - sum(penalty_i for checkpoint_i in violated at t)
```

如果连续多步 `progress_score` 不增长，且实际动作偏离当前应完成检查点，则进入规划失效候选。

### 5.2 遗漏必要步骤判定

触发条件：

```
required checkpoint not satisfied
AND later checkpoint satisfied or agent terminates
AND missing checkpoint is not optional
```

示例：

| 任务 | 遗漏 |
|------|------|
| 修改设置并保存 | 修改后未点击保存 |
| 下单 | 未选择规格或未确认订单 |
| 搜索并收藏 | 找到目标但未收藏 |

输出：

```json
{
  "type": "missing_required_step",
  "first_error_step": 5,
  "missing_checkpoints": ["点击保存"],
  "evidence": ["最终页面未出现保存成功提示", "路径中没有保存按钮相关动作"]
}
```

### 5.3 误解任务目标判定

触发条件：

```
actual_goal_embedding diverges from instruction_goal_embedding
OR key entity mismatch
OR forbidden_state appears and persists
```

关键实体包括商品名、联系人、文件名、设置项、App 功能模块、目标 tab、目标页面等。

建议规则：

| 信号 | 判定影响 |
|------|----------|
| 搜索词与用户目标实体不一致 | 强异常 |
| 页面核心实体与目标实体不一致 | 强异常 |
| ActionDescription 长期围绕无关功能 | 中强异常 |
| 达尔文整体意图 `nok` 且 wrong_reason 指向目标不一致 | 强异常 |

### 5.4 步骤顺序错误判定

将检查点建成有向无环图：

```
搜索商品 -> 进入详情 -> 选择规格 -> 加入购物车 -> 确认订单 -> 提交
```

触发条件：

```
checkpoint_j satisfied before checkpoint_i
AND checkpoint_i is a precondition of checkpoint_j
AND no valid shortcut rule exists
```

可接受的 shortcut 需要由 RAG 知识库或历史成功轨迹提供。例如某些 App 可以从搜索结果页直接加入购物车，此时不应误判未进入详情页。

### 5.5 提前终止判定

触发条件：

```
last_action in ["done", "finished"]
AND completion_score < threshold
AND at least one required checkpoint not satisfied
```

默认阈值：

| 任务类型 | 完成阈值 |
|----------|----------|
| 简单打开/查询 | `>= 0.8` |
| 多步骤交易/配置 | `>= 0.95` |
| 强约束任务 | 所有 required checkpoint 必须达成 |

达尔文 E2E `intention.label = nok` 且最后动作为 `finished` 时，优先归因为提前终止。

### 5.6 未能终止判定

触发条件：

```
completion_score >= threshold at step t
AND after t Agent continues more than m steps
AND subsequent actions do not improve completion_score
AND no explicit user requirement needs继续操作
```

默认 `m=2`。

该类异常常与重复动作、循环跳转耦合，应同时输出关联异常：

```json
{
  "type": "fail_to_terminate",
  "related_anomalies": ["repeated_action"],
  "completion_step": 8,
  "continued_until": 13
}
```

## 6. 达尔文能力的使用方式

达尔文现有能力适合作为规划失效判定的“语义证据源”，但不应作为唯一判断。

| 达尔文字段 | 使用方式 |
|------------|----------|
| `ab_pages_result` | 判断每一步页面跳转是否支持当前子目标 |
| `intention.label` | 判断整体任务是否达成 |
| `intention.wrong_reason` | 作为规划失败解释候选 |
| `llm_intention_step` / `vlm_intention_step` | 子目标完成度验证 |
| `*_intention_step_identity.wrong_steps` | 直接映射遗漏步骤 |
| `*_intention_step_identity.bug_steps` | 区分功能 bug 与规划失效 |

规划失效与功能 bug 的区分：

| 情况 | 归因 |
|------|------|
| Agent 未执行必要动作 | 规划失效 |
| Agent 执行了正确动作，但页面未按预期变化 | 功能 bug 或环境异常 |
| Agent 选错目标后继续执行 | 规划失效，可能伴随执行错误 |
| 页面加载失败导致无法继续 | 环境异常，规划失效只作次级标签 |

## 8. 与重复动作判定的关系

两类异常经常共现，但归因层级不同。

| 现象 | 首要归因 | 次级标签 |
|------|----------|----------|
| 目标已完成后继续点同一按钮 | 未能终止 | 重复动作 |
| 因不知道下一步而反复点击当前按钮 | 规划失效 | 重复动作 |
| 同一输入框反复输入错误关键词 | 误解任务目标 | 重复输入 |
| 页面无响应后多次重试 | 环境/功能异常 | 合理重复或低风险重复 |

综合报告中建议按根因排序：

```
规划失效 > 环境异常/功能 bug > 执行错误 > 重复动作/效率问题
```

但如果重复动作已经导致任务失败，也需要单独输出，便于效率和鲁棒性指标统计。

## 9. 技术穿刺计划

### 阶段一：基于显式步骤的规划判定

目标：先覆盖有 `step_level_instruction` 的测试用例。

交付：

- 解析 `step_level_instruction` 为检查点列表
- 对齐达尔文 `*_intention_step`
- 输出遗漏步骤、提前终止、整体意图未达成

### 阶段二：无显式步骤的自动分解

目标：覆盖只有 `instruction` 的真实任务。

交付：

- LLM 分解任务为子目标 DAG
- RAG 注入 App 流程知识
- VLM/规则检查每个子目标是否达成

### 阶段三：首错定位与归因

目标：从“结果失败”升级到“为什么失败、从哪一步开始失败”。

交付：

- `first_error_step` 定位
- 功能 bug / 环境异常 / 规划失效归因拆分
- 与重复动作、循环跳转检测结果融合

### 阶段四：标注集评估

建议构建最小标注集：

| 类型 | 数量 |
|------|------|
| 遗漏必要步骤 | 40 |
| 误解任务目标 | 30 |
| 步骤顺序错误 | 20 |
| 提前终止 | 30 |
| 未能终止 | 20 |
| 正常但路径不同 | 40 |

评估指标：

| 指标 | 目标 |
|------|------|
| 规划失效 Precision | >= 0.85 |
| 规划失效 Recall | >= 0.70 |
| 首错步骤误差 | <= 2 steps |
| 功能 bug 误归因为规划失效率 | <= 0.15 |

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 自动分解步骤不稳定 | 检查点错误导致误判 | 优先使用显式 `step_level_instruction` 和 RAG 模板 |
| App 存在多条合法路径 | 正常路径被误判为顺序错误 | 引入 shortcut 白名单和历史成功轨迹 |
| VLM 对最终状态误判 | 提前终止/遗漏步骤误判 | 核心检查点增加 OCR、布局和规则兜底 |
| 功能 bug 与规划失效混淆 | 归因错误 | 使用单步达尔文结果区分“动作没做”和“做了没生效” |
| 用户意图模糊 | 无法确定必要步骤 | 输出 `unknown` 或低置信度，不强判异常 |

---

## 11. 结论

规划失效判定的关键不是判断某一步动作是否正确，而是判断实际轨迹是否持续推进用户意图。落地上应优先从显式步骤和达尔文 E2E 子步骤结果切入，建立“子目标检查点 -> 实际状态进展 -> 终止时机”的判定链路，再逐步扩展到无显式步骤的自动任务分解和首错归因。
