# GUI Agent 测试平台 — 轨迹状态提取技术方案

> 子模块：基于动态滑动窗口 + UI 相似性理解的任务执行状态提取
> 编制时间：2026-07-13
> 上游需求：GUI Agent 测试平台 → 轨迹判定模块 → 关键状态准确性判定

---

## 1. 问题定位与方案边界

### 1.1 在测试平台中的位置

```
GUI Agent 执行
   │ 产出过程数据：截图序列 + 辅助信息(action/a11y/耗时/系统事件)
   ▼
┌──────────────────────────────────────┐
│  轨迹状态提取（本方案边界）            │
│  - 把原始帧序列压缩为「关键状态序列」  │
│  - 输出：StateGraph{S₀→S₁→…→Sₙ}      │
│        + 每个状态的语义标签与置信度    │
└──────────────────────────────────────┘   │
   ▼                                       │ 下游消费
┌──────────────────────────────────────┐   │
│  轨迹判定模块                          │ ◄─┘
│  - 关键状态准确性                      │
│  - 执行轨迹效率与合理性                │
│  - 异常行为识别                        │
└──────────────────────────────────────┘
   ▼
判定报告（准确率 / 冗余度 / 偏离度 / 回退次数 …）
```

### 1.2 核心难点

| 难点                  | 表现                                                  | 影响                             |
| --------------------- | ----------------------------------------------------- | -------------------------------- |
| **连续帧冗余**        | Agent 在同一页面停留时产生大量视觉上几乎相同的截图    | 直接用逐帧对比会淹没真正状态切换 |
| **视觉变化≠状态变化** | 加载动画、广告轮播、键盘弹出/收起都会引起大幅像素变化 | 单纯像素相似度会误判状态切换     |
| **状态变化≠视觉变化** | 列表选中态、单选切换、后台数据更新可能像素变化极小    | 纯视觉会漏判关键状态             |
| **动作粒度不均**      | 有的任务一步到位，有的在同一页面执行多次微操作        | 固定窗口无法适配                 |
| **回退/重试**         | Agent 可能回退到之前状态，不是严格线性                | 状态序列需要支持环/重访          |
| **异步加载**          | 网络延迟导致"看似未变化"但实际正在加载                | 需区分"稳定态"与"过渡态"         |

### 1.3 方案设计原则

1. **多模态融合**：截图(视觉) + a11y tree(结构) + action log(行为) + 系统事件(时序) 四路信号联合判定，任一通道噪声不致命。
2. **动态自适应**：窗口大小根据 UI 变化剧烈程度动态伸缩，而非固定步长。
3. **分层抽象**：像素级 → 结构级 → 语义级 三层相似性，由低到高逐级过滤。
4. **可解释**：每个被识别的关键状态都要输出"为什么被选为关键状态"的证据链。
5. **与下游对齐**：输出的状态图直接对接判定模块的"关键状态准确性/效率/合理性"三个指标。

## 3. 关键技术详解

### 3.1 数据采集与对齐

#### 3.1.1 输入数据规范

每个 Agent 执行轨迹 (`Trajectory`) 包含一个有序的步 (`Step`) 列表：

```python
@dataclass
class Step:
    # —— 视觉 ——
    screenshot: bytes              # PNG/JPEG 原始截图
    screenshot_meta: dict          # {w, h, format, capture_ts}

    # —— 结构（来自 a11y tree / DOM / UIAutomator2）——
    ui_tree: dict                  # 树形 JSON：节点含 type/bounds/text/cls/clickable/...
    ui_tree_hash: str              # 结构指纹（见 3.2.2）

    # —— 行为 ——
    action: dict                   # {type: tap/swipe/input/scroll/wait/..., target, params, reasoning}
    action_target_bbox: tuple      # (x, y, w, h) 动作目标区域
    action_result: str             # success / fail / timeout / noop

    # —— 时序 ——
    step_index: int
    timestamp_ms: int              # 动作执行时刻
    pre_delay_ms: int              # 动作前等待
    post_delay_ms: int             # 动作后等待
    duration_ms: int               # 动作执行耗时

    # —— 系统事件（可选）——
    system_events: list            # [Toast, Dialog, KeyboardShow/Hide, PageTransition, ...]
```

#### 3.1.2 多源对齐

| 信号       | 采集频率                                      | 对齐方式                            |
| ---------- | --------------------------------------------- | ----------------------------------- |
| 截图       | 每步动作前后各 1 帧（pre-frame + post-frame） | 以 `step_index` 为主键              |
| a11y tree  | 与截图同步                                    | 同上                                |
| Action log | 动作执行瞬间                                  | 同上                                |
| 系统事件   | 异步事件流                                    | 以 `timestamp_ms` 归属到最近的 step |

> **关键约定**：每个 Step 产生两帧——`pre_frame`（动作前状态）和 `post_frame`（动作后状态）。状态提取主要基于 post_frame 序列（代表"每个动作完成后的世界状态"），pre_frame 用于过渡态检测。

### 3.2 预处理与多层特征提取

#### 3.2.1 视觉特征层（像素级）

**目的**：快速、廉价地计算帧间视觉差异，用于滑动窗口的第一道过滤。

| 特征                        | 算法                                       | 用途                                   | 计算成本       |
| --------------------------- | ------------------------------------------ | -------------------------------------- | -------------- |
| **pHash** (Perceptual Hash) | DCT → 中值哈希，64bit                      | 粗粒度视觉相似性，快速过滤             | 极低           |
| **dHash** (Difference Hash) | 水平差分哈希                               | pHash 补充，对水平变化敏感             | 极低           |
| **颜色直方图**              | HSV 空间 3D 直方图 (bins: H=16,S=8,V=8)    | 全局色调变化检测（主题切换、深色模式） | 低             |
| **SSIM** (结构相似性)       | 11×11 高斯窗口                             | 局部结构变化，比像素 diff 更鲁棒       | 中             |
| **区域 IoU**                | 按动作目标 bbox 划分 ROI，计算 ROI 内 SSIM | 聚焦"动作影响区域"的变化               | 中             |
| **CLIP 嵌入**               | CLIP-ViT-B/32 或移动端 MobileCLIP          | 语义级视觉相似性（下游融合用）         | 高（可选 GPU） |

**视觉差异度量**：

```
Δ_visual(i, j) = α · (1 - Hamming(pHashᵢ, pHashⱼ) / 64)
              + β · (1 - hist_intersection(Hᵢ, Hⱼ))
              + γ · (1 - SSIM(frameᵢ, frameⱼ))
              + δ · (1 - SSIM(ROIᵢ, ROIⱼ))    # 动作影响区域

典型权重：α=0.2, β=0.15, γ=0.35, δ=0.30
```

> **设计要点**：pHash/dHash 作为"快速预筛"——如果两帧 pHash 海明距离 ≤ 3，直接判定为视觉相似，跳过后续昂贵计算。这是一个级联过滤（cascade）设计。

#### 3.2.2 结构特征层（UI 树级）

**目的**：捕获视觉上不易察觉但结构上发生的状态变化（如单选切换、选中态、数据加载完成）。

**UI 树指纹 (Structural Fingerprint)**：

```python
def compute_tree_fingerprint(ui_tree: dict) -> str:
    """
    对 UI 树做结构哈希：
    1. 过滤掉动态噪声节点（时间戳、广告 ID、动画 View）
    2. 对每个节点提取 (type, bounds_area_normalized, text_hash, clickable,
                       is_focusable, is_checked, is_selected)
    3. 对树做后序遍历，拼接节点特征后做 SHA-256
    """
    # 过滤规则
    NOISE_PATTERNS = [
        r"^\d{1,2}:\d{2}$",         # 时钟
        r"^[\d,.]+%?$",              # 纯数字（动态计数器）
        r"ad[_-]?container",         # 广告容器
        # ...
    ]
    filtered = filter_tree(ui_tree, NOISE_PATTERNS)
    # 归一化 bounds（除以屏幕尺寸，转为 0~1 比例）
    normalized = normalize_bounds(filtered, screen_w, screen_h)
    # 后序遍历哈希
    return postorder_hash(normalized)
```

**结构差异度量**：

```
Δ_struct(i, j) = 
    0.0                         if fingerprint_i == fingerprint_j
    tree_edit_distance(Tᵢ, Tⱼ) / max(|Tᵢ|, |Tⱼ|)   otherwise
```

其中 `tree_edit_distance` 使用简化版 Zhang-Shasha 算法，节点匹配时只看 type + 归一化位置（不做精确 text 匹配，允许动态文本差异）。

**关键子指标**：

- `node_added_ratio`：新增节点数 / 总节点数 → 检测"新内容出现"（页面跳转、弹窗）
- `node_removed_ratio`：移除节点数 / 总节点数 → 检测"内容消失"（关闭弹窗、返回）
- `text_changed_ratio`：文本变化节点数 / 总节点数 → 检测"数据更新"（搜索结果刷新）
- `state_attr_changed_ratio`：状态属性变化节点数（checked/selected/enabled）/ 总节点数 → 检测"选中态切换"

#### 3.2.3 行为特征层（Action 级）

**目的**：用 Agent 自身的动作类型辅助判断状态边界。某些动作天然暗示状态切换：

| Action Type | 状态切换先验概率 P(switch|action) | 说明 |
|-------------|----------------------------------|------|
| `navigate_back` | 0.85 | 返回操作大概率引起页面切换 |
| `tap`(导航类元素) | 0.70 | tab bar、底部导航、菜单项 |
| `tap`(按钮) | 0.55 | 一般按钮可能触发弹窗/页面跳转 |
| `input` | 0.15 | 输入通常不改变页面结构 |
| `scroll` | 0.10 | 滚动只在同一页面内移动 |
| `swipe` | 0.20 | 滑动可能是翻页也可能只是滚动 |
| `wait` | 0.05 | 等待动作本身不改变状态 |
| `long_press` | 0.40 | 可能触发上下文菜单 |

**动作目标语义分类**：用 a11y tree 中目标节点的 `class` + `text` + `parent_class` 判断动作目标是否属于"导航类"（如 `BottomNavigationItemView`、`TabLayout$TabView`、`MenuItem`）。

#### 3.2.4 时序特征层

| 特征                        | 计算方式                          | 用途                             |
| --------------------------- | --------------------------------- | -------------------------------- |
| `inter_step_interval`       | post_frameᵢ.ts - post_frameᵢ₋₁.ts | 过长的间隔可能暗示异步加载或卡顿 |
| `action_duration`           | action 执行耗时                   | 异常长的动作可能失败             |
| `visual_stabilization_time` | post_frame 到下一帧不再变化的时间 | 判断"是否已稳定"                 |
| `event_density`             | step 内系统事件数量               | 高事件密度可能暗示状态切换       |

### 3.3 动态滑动窗口引擎

这是方案的核心——**用动态大小的滑动窗口扫描帧序列，在窗口内判断"是否发生了状态切换"，并根据变化剧烈程度自动调整窗口大小**。

#### 3.3.1 窗口模型

```
Frame 序列:  F₀  F₁  F₂  F₃  F₄  F₅  F₆  F₇  F₈  F₉  F₁₀ ...
              ├──── 窗口 W₁ ──┤
                    ↑ 窗口内检测到 F₃ 处发生状态切换
                                  ├──── 窗口 W₂ ──┤
                                       ↑ 窗口内无切换，扩张窗口
                                  ├──────── 窗口 W₂'(扩张后) ────────┤
                                                    ↑ F₈ 处发生切换
```

#### 3.3.2 窗口状态机

```
                    ┌──────────────┐
            ┌──────▶│  EXPANDING   │◀───── 初始状态
            │       │ 窗口逐步增大  │
            │       │ 每步检查 Δ   │
            │       └──────┬───────┘
            │              │ 检测到 Δ > θ_switch
            │              ▼
            │       ┌──────────────┐
            │       │  CONFIRMING  │
            │       │ 回看 k 帧确认 │
            │       └──────┬───────┘
            │              │ 确认切换
            │              ▼
            │       ┌──────────────┐
            │       │  SPLITTING   │  → 输出旧状态 Sₖ
            │       │ 在切换点分割  │    开始新状态 Sₖ₊₁
            │       └──────┬───────┘
            │              │
            └──────────────┘
            继续以最小窗口开始扩张

            若 CONFIRMING 阶段未确认：
            ┌──────────────┐
            │  SHRINKING   │  → 缩小窗口继续扫描
            │  缩小窗口     │
            └──────────────┘
                   │
                   ▼ 回到 EXPANDING
```

#### 3.3.3 窗口大小动态调整策略

```python
class AdaptiveWindow:
    def __init__(self):
        self.min_size = 2          # 最少看 2 帧
        self.max_size = 15         # 最多回看 15 帧
        self.current_size = 2
        self.growth_rate = 1.5     # 扩张倍率
        self.shrink_rate = 0.7     # 收缩倍率

    def next_window_size(self, recent_deltas: list[float]) -> int:
        """
        根据最近的 Δ 序列动态调整窗口大小。

        - 如果最近几帧 Δ 都很低（稳定态）→ 扩张窗口，减少计算
        - 如果最近几帧 Δ 波动大（过渡态）→ 缩小窗口，提高精度
        - 如果最近几帧 Δ 先低后高（可能是切换点）→ 保持窗口，进入确认
        """
        if len(recent_deltas) < 2:
            return self.min_size

        recent = recent_deltas[-5:]
        trend = sum(recent[-3:]) / 3 - sum(recent[:2]) / 2  # 趋势：上升/下降/平稳

        if max(recent) < 0.15:
            # 非常稳定，加速扩张
            self.current_size = min(
                int(self.current_size * self.growth_rate) + 1,
                self.max_size
            )
        elif max(recent) > 0.5 and trend > 0.2:
            # 波动大且上升趋势，可能正在切换，缩小窗口精确定位
            self.current_size = max(
                int(self.current_size * self.shrink_rate),
                self.min_size
            )
        # else: 保持当前大小

        return self.current_size
```

#### 3.3.4 状态切换判据融合

在窗口内任意位置，综合四层特征判定是否发生状态切换：

```python
def detect_state_transition(
    frame_i: Frame, frame_j: Frame, step_i: Step, step_j: Step
) -> TransitionVerdict:
    """
    判定 frame_i → frame_j 之间是否发生了"有意义的状态切换"。
    返回 (is_transition: bool, confidence: float, evidence: list[str])
    """
    # —— 层 1：视觉差异（快速预筛）——
    phash_dist = hamming(frame_i.phash, frame_j.phash)
    if phash_dist <= 2:
        # 视觉几乎相同 → 看结构层是否有隐式状态变化
        if step_j.action_type in HIGH_SWITCH_PRIOR_ACTIONS:
            return maybe_transition(confidence=0.3, reason="视觉相同但动作暗示切换")
        return no_transition()

    # —— 层 2：结构差异 ——
    struct_delta = tree_edit_distance_norm(frame_i.ui_tree, frame_j.ui_tree)
    node_added = node_added_ratio(frame_i.ui_tree, frame_j.ui_tree)
    node_removed = node_removed_ratio(frame_i.ui_tree, frame_j.ui_tree)

    # —— 层 3：行为先验 ——
    action_prior = ACTION_SWITCH_PRIOR.get(step_j.action_type, 0.3)

    # —— 层 4：时序 ——
    time_gap = frame_j.ts - frame_i.ts
    has_loading_event = any(e.type == "PageTransition" for e in step_j.system_events)

    # —— 融合判定 ——
    # 视觉变化大 + 结构变化大 → 高置信度切换
    if struct_delta > 0.35 and (node_added > 0.2 or node_removed > 0.2):
        return transition(confidence=0.85, evidence=["结构大幅变化", f"新增节点比例{node_added:.0%}"])

    # 视觉变化大但结构变化小 → 可能是动画/加载/键盘弹出
    if struct_delta < 0.10:
        # 检查是否是已知"伪切换"
        if is_keyboard_event(step_j.system_events):
            return no_transition(reason="键盘弹出/收起，非状态切换")
        if is_animation_or_loading(frame_i, frame_j):
            return no_transition(reason="动画/加载效果，非状态切换")
        # 视觉变了但结构没变 → 可能是主题切换或图片加载完成
        # 降级为低置信度切换
        return maybe_transition(confidence=0.4, reason="视觉变化但结构稳定")

    # 视觉变化小 + 结构变化大 → 隐式状态变化（选中态、数据更新）
    if struct_delta > 0.20 and phash_dist <= 5:
        if state_attr_changed_ratio(frame_i.ui_tree, frame_j.ui_tree) > 0.05:
            return transition(confidence=0.70, evidence=["状态属性变化", "视觉变化小但选中/聚焦态改变"])
        if text_changed_ratio(frame_i.ui_tree, frame_j.ui_tree) > 0.10:
            return transition(confidence=0.65, evidence=["文本内容变化", "数据可能已更新"])

    # 动作先验加权
    if action_prior > 0.6 and (struct_delta > 0.15 or phash_dist > 5):
        return transition(confidence=0.60, evidence=[f"高切换先验动作({step_j.action_type})"])

    # 默认：综合加权
    score = (
        0.30 * min(phash_dist / 10, 1.0) +
        0.35 * struct_delta +
        0.20 * action_prior +
        0.15 * (1.0 if has_loading_event else 0.0)
    )
    if score > 0.45:
        return transition(confidence=score, evidence=[f"综合评分{score:.2f}"])
    return no_transition()
```

#### 3.3.5 过渡态与稳定态区分

不是所有状态切换都应被记录为"关键状态"。需要区分：

| 状态类型                  | 特征                                                    | 处理方式                           |
| ------------------------- | ------------------------------------------------------- | ---------------------------------- |
| **稳定态 (Stable)**       | UI 在可观察时间内不再变化                               | 记录为关键状态候选                 |
| **过渡态 (Transient)**    | UI 正在变化中（加载中、动画中、滚动中）                 | 不记录为关键状态，但标记"正在过渡" |
| **中间态 (Intermediate)** | 短暂出现但随后消失的 UI（如 Toast、Snackbar、临时弹窗） | 记录为"临时状态"，不作为主状态     |

**稳定态判定**：在切换点之后，连续观察 `N_stable` 帧（默认 N=3），如果这 N 帧之间的 Δ 都低于阈值 `θ_stable`（默认 0.12），则判定为稳定态。

```python
def is_stable_state(window: list[Frame], start_idx: int) -> bool:
    """检查从 start_idx 开始连续 N 帧是否足够稳定。"""
    N = 3
    for i in range(start_idx, min(start_idx + N, len(window) - 1)):
        delta = compute_visual_delta(window[i], window[i + 1])
        struct_delta = compute_struct_delta(window[i], window[i + 1])
        if delta > 0.12 or struct_delta > 0.08:
            return False
    return True
```

### 3.4 状态聚合与图构建

#### 3.4.1 状态表示

每个聚合后的关键状态：

```python
@dataclass
class KeyState:
    state_id: str                    # UUID
    label: str                       # 语义标签，如 "搜索结果页"、"购物车页"
    representative_frame: bytes      # 最具代表性的截图（选窗口内最接近均值的帧）
    representative_step_idx: int     # 对应的 step

    # —— 组成帧 ——
    frame_indices: list[int]         # 属于该状态的所有帧索引
    step_indices: list[int]          # 属于该状态的所有 step 索引

    # —— 特征摘要 ——
    ui_tree_fingerprint: str         # 该状态的结构指纹（众数）
    clip_embedding: np.ndarray       # 该状态的 CLIP 嵌入（帧均值）
    action_summary: dict             # 该状态内 Agent 执行的动作统计

    # —— 时间信息 ——
    first_seen_ts: int
    last_seen_ts: int
    total_duration_ms: int           # Agent 在该状态停留的总时长

    # —— 置信度与证据 ——
    confidence: float                # 被判定为关键状态的置信度
    evidence: list[str]              # 判定依据列表

    # —— 状态类型 ——
    state_type: str                  # "stable" | "intermediate" | "transient"

    # —— 语义标注（可选，由下游 LLM 填充）——
    semantic_description: str        # "用户正在查看商品详情页，商品名为XXX"
    expected_state_match: str        # 如果有预期状态定义，匹配到的预期状态 ID
```

#### 3.4.2 状态图 (StateGraph) 构建

```python
@dataclass
class StateGraph:
    states: list[KeyState]
    transitions: list[StateTransition]
    start_state_id: str
    end_state_id: str

@dataclass
class StateTransition:
    from_state: str                  # state_id
    to_state: str                    # state_id
    trigger_action: dict             # 触发切换的动作
    trigger_step_idx: int
    transition_type: str             # "forward" | "back" | "retry" | "loop" | "error"
    confidence: float
```

**关键能力——环/重访检测**：

当新提取的状态与已有状态高度相似时（CLIP 余弦相似度 > 0.92 且结构指纹相同），不创建新状态，而是添加一条到已有状态的转移边，标记为 `transition_type = "back"` 或 `"loop"`。这对下游判定"Agent 是否在兜圈子"至关重要。

#### 3.4.3 状态去重与合并

```python
def merge_states(states: list[KeyState]) -> list[KeyState]:
    """
    对提取出的状态做后处理：
    1. 视觉+结构双重相似的状态合并
    2. 过短的中间态（duration < 500ms）如果夹在两个相同状态之间 → 视为噪声删除
    3. 连续相同的状态合并
    """
    merged = []
    for s in states:
        if not merged:
            merged.append(s)
            continue
        prev = merged[-1]
        # 同状态：结构指纹相同 + CLIP 相似度 > 0.90
        if (prev.ui_tree_fingerprint == s.ui_tree_fingerprint and
            cosine_sim(prev.clip_embedding, s.clip_embedding) > 0.90):
            # 合并
            prev.frame_indices.extend(s.frame_indices)
            prev.step_indices.extend(s.step_indices)
            prev.last_seen_ts = s.last_seen_ts
            prev.total_duration_ms += s.total_duration_ms
        else:
            merged.append(s)

    # 过滤过短的中间态噪声
    filtered = [s for s in merged
                if s.state_type != "intermediate" or s.total_duration_ms >= 500]
    return filtered
```

### 4.3 关键参数与调优建议

| 参数                        | 默认值      | 调优方向                                        |
| --------------------------- | ----------- | ----------------------------------------------- |
| `θ_switch` (切换阈值)       | 0.45        | 过高会漏检，过低会误检；建议按 App 类型分组调优 |

| pHash 预筛阈值              | ≤3 视为相似 | 截图分辨率低时降低到 2                          |
| 中间态最短持续时间          | 500ms       | 可按 App 响应速度调整                           |

## 6. 风险与缓解

| 风险                                                | 影响                                   | 缓解措施                                                     |
| --------------------------------------------------- | -------------------------------------- | ------------------------------------------------------------ |
| **a11y tree 不可用**（如游戏应用、Canvas 绘制界面） | 结构特征层失效                         | 降级为纯视觉模式：用 CLIP 嵌入 + OCR 文本替代结构特征        |
| **CLIP 推理成本高**                                 | 大规模测试时成本过高                   | 级联设计：仅对 pHash/SSIM 无法判定的"边界帧"调用 CLIP        |
| **阈值不适配不同 App**                              | 电商类 App 和工具类 App 的最佳阈值不同 | 按品类分组维护阈值配置，支持运行时热更新                     |
| **VLM 标签生成延迟**                                | 语义标签阶段成为瓶颈                   | 异步生成标签，状态图先输出，标签后填充；或对小任务跳过 VLM   |
| **Agent 回退导致状态图出现环**                      | 下游判定逻辑需要处理非线性图           | 状态图数据结构支持环；判定模块显式处理 back/loop 类型转移    |
| **异步加载导致的"假稳定"**                          | Agent 看到页面以为稳定但数据还在加载   | 检查系统事件中是否有 `isLoading` / `ProgressBar` 残留；延长稳定观察帧数 |

## 7. 方案亮点总结

1. **级联过滤设计**：pHash 预筛 → SSIM 验证 → 结构层确认 → CLIP 语义层兜底，在保证准确率的同时大幅降低计算成本。
2. **四维融合判据**：视觉 + 结构 + 行为 + 时序四路信号联合判定状态切换，任何单一通道的噪声（如动画、键盘弹出）不会导致误判。
3. **动态窗口自适应**：窗口大小根据 UI 变化剧烈程度自动伸缩，稳定区域加速通过，过渡区域精细扫描，兼顾效率和精度。
4. **过渡态/稳定态/中间态三分**：不是简单二分"状态变/不变"，而是区分"正在变"、"已稳定"和"短暂出现"三类，更贴合 GUI 实际情况。
5. **状态图而非状态序列**：支持环、重访、回退，真实反映 Agent 的执行路径（包括犯错和纠正），为下游"效率与合理性判定"提供丰富信号。
6. **可解释性**：每个关键状态附带证据链（为什么被判定为关键状态）和置信度，便于测试人员审计和调优。
