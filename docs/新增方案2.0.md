# GUI Agent 测试平台 — 轨迹状态提取技术方案 v2.0

> 子模块：基于 actionPurpose 语义标注 + clearRes OCR 树的状态提取与对齐
> 编制时间：2026-07-14
> v1.0 主要变更：用实际数据源（clearRes + actionPurpose）替代假定的 a11y tree；引入 Intent-vs-Execution 双层判定对接

---

## 1. 问题定位与方案边界

### 1.1 前置认知

经过对实际数据管线（`utg.json` + `clearRes.gzip` + 截图序列）的分析，项目判定体系已确定为一个双层对比模型：

```
  ┌──────────────────────────────────────────────────┐
  │                Intent 层（规划判定）               │
  │  模块A RAG 检查点  ←对比→  actionPurpose        │
  │  "应该做什么"             "Agent 自述想做什么"     │
  │         ↓ 意图偏离 = 规划失效                      │
  ├──────────────────────────────────────────────────┤
  │              Execution 层（执行判定）              │
  │  本方案产出 StateGraph  ←对比→  Darwin VLM 截图   │
  │  "实际经过了哪些UI状态"      "截图证明结果对吗"    │
  │         ↓ 状态偏差 = 执行异常                      │
  └──────────────────────────────────────────────────┘
```

本方案 v2.0 的职责：**填充 Execution 层的输入**——将 Agent 的原始截图序列 + 辅助数据压缩为结构化状态图，供下游判定模块做对比。

### 1.2 实际可用数据源（与 v1.0 的关键差异）

| 数据源 | v1.0 假设 | v2.0 实际 | 影响 |
|--------|----------|----------|------|
| 结构信号 | Android a11y tree | `clearRes.json` OCR 树 | 已有 text/type/bounds/父子关系，且含 icon 标签关联 |
| 行为信号 | action log | `directives` + `actionPurpose` | `actionPurpose` 天然标注了状态切换意图 |
| 视觉信号 | pre/post frame | `catchDataTurnId*.jpg` | 一致 |
| 元素名 | a11y node.text | `clearRes` 兄弟 text 节点 | icon 元素的标签可通过树结构关联到同级 text 节点 |

### 1.3 核心难点（v2.0 重定义）

| 难点 | v1.0 级别 | v2.0 降维 |
|------|----------|----------|
| 连续帧冗余 | 需要四层融合判据 | `actionPurpose` 语义相同 = 大概率同状态，pHash 验证即可 |
| 视觉变化≠状态变化 | 需要结构层验证 | `clearRes` OCR 树不变 = 结构未变，排除动画/键盘等伪切换 |
| 状态变化≠视觉变化 | 需要 CLIP 语义层 | `clearRes` text 节点变化即可捕获选中态/数据更新 |
| 回退/重试 | 需要环检测算法 | `actionPurpose` + StateGraph 指纹去重直接标注 back/loop |

## 3. 关键技术详解

### 3.1 数据对齐与解析

#### 3.1.1 输入数据规范（基于实际源文件）

每个任务目录下的可用文件和对齐关系：

```
<task_uuid>/
├── utg.json              → stepData(动作序列) + nodes(截图引用) + edges(事件)
├── clearRes.gzip         → imageId + rawPage(OCR树) + actionPurpose(每步意图)
└── catchDataTurnIdN/
    └── temp_image-screenshot-origin.jpg
```

**对齐键**：

| 信号 | 来源 | 对齐方式 |
|------|------|---------|
| 截图 | `nodes[].image` → `catchDataTurnIdN/` | `stepId` ↔ `image` URL |
| OCR 树 | `clearRes.json` rawPage | `imageId` ↔ 截图 URL |
| 动作坐标 | `directives.params.points` / `node.bounds` | `stepId` |
| 动作意图 | `clearRes.actionPurpose` | 按 `stepId` 顺序对应 |
| 事件 | `edges[].events[]` | `from`/`to` ↔ `node.id` |

#### 3.1.2 Step 数据结构

```python
@dataclass
class AlignedStep:
    step_id: str
    # —— 视觉 ——
    screenshot_path: str            # 截图本地路径
    # —— OCR 结构 ——
    ocr_tree: dict                  # clearRes.json 的 rawPage 中对应页面的 OCR 节点树
    # —— 动作 ——
    action_type: str                # click / scroll / type / open_app / clarify / ...
    action_start_box: list[int]     # 触点坐标 [x, y]
    action_end_box: list[int]       # 多指滑动终点（可选）
    action_target: str              # 控件名称（OCR text fallback 见 3.2.3）
    action_purpose: str             # Agent 自述意图
    # —— 时序 ——
    cost_time: int                  # 耗时 ms
    timestamp: int                  # 绝对时间戳
```

### 3.2 状态边界标注（actionPurpose 驱动）

这是 v2.0 与 v1.0 最大的简化点。

#### 3.2.1 actionPurpose 的语义分类

Agent 每步的 `actionPurpose` 包含对自身行为的描述，可以自动分类：

| actionPurpose 语义模式 | 推断 | 示例 |
|-----------------------|------|------|
| "进入..."、"打开..."、"跳转到..." | 明确状态切换 | "点击设置以进入设置首页" |
| "向下滑动寻找..."、"滚动查看..." | 页面内探索（当前状态延续） | "向下滑动寻找隐私选项" |
| "输入..."、"填写..." | 表单交互（当前状态延续） | "在搜索框输入定时开关机" |
| "任务完成"、"手动操作" | 终端状态 | "当前页面需要手动操作" |

**推断规则**：

```python
STATE_TRANSITION_PATTERNS = [
    (r"进入|打开|跳转|返回|退出", "state_transition"),
    (r"滑动|滚动|翻页|查看", "in_state_exploration"),
    (r"输入|填写|选择|勾选", "in_state_interaction"),
    (r"完成|结束|手动|无法", "terminal"),
]

def classify_purpose(action_purpose: str) -> str:
    for pattern, label in STATE_TRANSITION_PATTERNS:
        if re.search(pattern, action_purpose):
            return label
    return "in_state_exploration"  # 默认页面内
```

  Step2: "向下滑动寻找隐私和安全选项"        → in_state_exploration
  Step3: "继续向下滑动寻找密码保险箱"        → in_state_exploration

  Step5: "向下滑动至密码保险箱选项可见"       → in_state_exploration
  Step6: "点击密码保险箱以管理密码"           → state_transition ← 状态边界候选

边界候选点: Step1 → Step4 → Step6 → Step7

#### 3.2.3 控件名称解析（clearRes 树关联）

对于 `actionPurpose` 中提到的控件（如"隐私和安全"、"密码保险箱"），需要从 `clearRes` OCR 树中提取对应的控件名称。

**问题**：`directives.params.node.text` 对 icon 类型元素为空。

**解决**：在 `clearRes` OCR 树中查找动作坐标所在的节点，然后找同级 text 节点：

```python
def resolve_target_name(ocr_tree: dict, touch_point: list[int]) -> str:
    """
    给定 OCR 树和触点坐标，返回控件名称。
    
    1. 查找触点所在的叶子节点
    2. 如果该节点 type=icon 且 text 为空：
       → 查找同级 text 子节点（如 icon 下方的标签文字）
    3. 如果该节点的父节点是 layout 且有多个 text 子节点：
       → 返回距离触点最近的 text
    """
    node = find_node_at_point(ocr_tree, touch_point)
    if not node:
        return ""
    
    # 优先取自身 text
    if node.get("text", "").strip():
        return node["text"]
    
    # icon 元素：查找同级 text
    if node.get("type") == "icon":
        parent = find_parent(ocr_tree, node["id"])
        if parent:
            texts = [s for s in parent.get("subNodes", [])
                     if s.get("type") == "text" and s.get("text", "").strip()]
            if texts:
                # 返回距离触点最近的 text 节点
                return closest_text(texts, touch_point)
    
    # 兜底：用 bounds 区域内的任意 text
    return find_text_in_bounds(ocr_tree, node.get("bounds", []))
```

#### 3.2.4 边界验证（pHash + OCR 指纹）

`actionPurpose` 标注的边界候选需要通过视觉和结构信号验证：

```python
def verify_state_boundary(step_i: AlignedStep, step_j: AlignedStep) -> bool:
    """
    验证 step_i → step_j 是否真的是状态切换。
    
    验证逻辑（级联，早期退出）：
    1. actionPurpose 说"state_transition" → 初步候选
    2. pHash 海明距离 > 3 → 视觉确实变了
    3. clearRes OCR 树指纹不同 → 结构确实变了
    4. 排除伪切换（键盘弹出、动画、Toast）
    
    只要 2 或 3 任一满足且不触发伪切换 → 确认状态边界
    """
    # 层 1: 快速视觉预筛
    if hamming_distance(step_i.screenshot_path, step_j.screenshot_path) <= 3:
        # 视觉没变，检查是否有隐式结构变化（选中态等）
        if ocr_fingerprint(step_i.ocr_tree) == ocr_fingerprint(step_j.ocr_tree):
            return False  # 完全没变，假边界
        # 结构变了但视觉没变 → 可能是选中态/数据更新，算小状态变化
        return True
    
    # 层 2: 排除伪切换
    if is_keyboard_event(step_j):  # 键盘弹出
        return False
    if is_toast_or_snackbar(step_i.ocr_tree, step_j.ocr_tree):  # 短暂弹出
        return False
    
    # 层 3: 确认
    return True
```

**OCR 树指纹**（轻量版，无需 tree edit distance）：

```python
def ocr_fingerprint(ocr_tree: dict) -> str:
    """
    计算 OCR 树的结构指纹。
    提取所有可见 text 节点的 (type, bounds_normalized, text_hash) 元组，
    排序后做 SHA-256。
    """
    text_nodes = []
    def collect(node):
        if node.get("text", "").strip():
            bounds = node.get("bounds", [])
            normalized = [b / 1280 for b in bounds[:4]] if bounds else []
            text_nodes.append((
                node.get("type", ""),
                tuple(normalized),
                hash(node["text"]),
            ))
        for sub in node.get("subNodes", []):
            collect(sub)
    collect(ocr_tree)
    text_nodes.sort()
    return hashlib.sha256(str(text_nodes).encode()).hexdigest()[:16]
```

### 3.3 状态聚合与图构建

#### 3.3.1 状态表示

```python
@dataclass
class KeyState:
    state_id: str                    # 如 "s_0"
    state_type: str                  # stable | transient | intermediate | terminal
    # —— 标识 ——
    label: str                       # 语义标签，如 "设置首页"、"隐私设置页"
    ocr_fingerprint: str            # OCR 树指纹（去重用）
    # —— 覆盖范围 ——
    step_range: tuple[int, int]     # [起始 step, 结束 step)
    action_summary: list[str]       # 该状态内的 actionPurpose 摘要
    # —— 时间 ——
    duration_ms: int                # 停留时长
    # —— 证据 ——
    boundary_evidence: list[str]    # 为什么被判定为关键状态
    confidence: float
```

#### 3.3.2 状态图构建

```python
@dataclass
class StateGraph:
    states: list[KeyState]
    transitions: list[StateTransition]
    start_state_id: str
    end_state_id: str

@dataclass
class StateTransition:
    from_state: str
    to_state: str
    trigger_step_idx: int           # 触发切换的步骤
    trigger_action_purpose: str     # Agent 自述的切换原因
    transition_type: str            # forward | back | loop | error
```

#### 3.3.3 环/回退检测

当新提取的状态的 OCR 指纹与已存在状态相同（且时间上不连续）时，标记为回退：

```python
def detect_loop_or_back(
    current_state: KeyState,
    existing_states: list[KeyState],
    step_idx: int,
) -> str:  # "forward" | "back" | "loop"
    for prev in existing_states:
        if prev.ocr_fingerprint == current_state.ocr_fingerprint:
            if prev.step_range[1] == step_idx - 1:
                return "forward"      # 连续，正常前进
            else:
                return "back"         # 非连续，回退了
    return "forward"
```

#### 3.3.4 状态去重与合并

```python
def merge_states(states: list[KeyState]) -> list[KeyState]:
    """合并连续相同 OCR 指纹的状态，过滤过短的中间态。"""
    merged = []
    for s in states:
        if not merged:
            merged.append(s)
            continue
        prev = merged[-1]
        if prev.ocr_fingerprint == s.ocr_fingerprint:
            # 合并 step_range 和时间
            prev.step_range = (prev.step_range[0], s.step_range[1])
            prev.duration_ms += s.duration_ms
            prev.action_summary.extend(s.action_summary)
        else:
            merged.append(s)
    
    # 过滤 duration < 500ms 且夹在两个相同状态之间的中间态
    filtered = []
    for i, s in enumerate(merged):
        if s.state_type == "intermediate" and s.duration_ms < 500:
            if i > 0 and i < len(merged) - 1:
                if merged[i-1].ocr_fingerprint == merged[i+1].ocr_fingerprint:
                    continue
        filtered.append(s)
    return filtered
```

### 3.4 语义标签生成

状态图构建完成后，为每个状态生成可读标签。v2.0 优先使用 `actionPurpose` + `clearRes` text 信息，VLM 作为可选增强。

#### 3.4.1 规则标签（零成本）

```python
def generate_label_rule(state: KeyState) -> str:
    """
    基于 actionPurpose + OCR text 生成状态标签。
    
    规则：
    1. 取该状态首步的 actionPurpose → 提取目标页面名
    2. 取 OCR 树中高频出现的页面标识性 text
    3. 组合："{app} {页面}" 格式
    """
    # 从 actionPurpose 提取目标
    first_purpose = state.action_summary[0] if state.action_summary else ""
    target_match = re.search(r"(进入|打开)(.+?)(页面|应用|设置|功能|界面)?", first_purpose)
    if target_match:
        return target_match.group(2).strip()
    
    # 从 OCR 树取顶部高频 text（标题栏等）
    top_texts = extract_top_level_texts(state.ocr_fingerprint)
    if top_texts:
        return top_texts[0]
    
    return "未知状态"
```

#### 3.4.2 VLM 标签（按需调用）

仅对规则标签置信度低的"边界状态"调用 VLM，大幅减少调用量：

```python
async def generate_label_vlm(state: KeyState, screenshot_path: str) -> str:
    """VLM 生成语义标签，仅对低置信度状态调用。"""
    if state.confidence > 0.7:
        return generate_label_rule(state)  # 高置信度直接用规则
    
    # 低置信度：调 VLM
    prompt = f"用一句话描述这个界面：页面类型、关键内容、与什么任务相关"
    return await vlm_generate(prompt, image=screenshot_path)
```

### 4.2 下游判定模块接口

```
StateGraph
  │
  ├──▶ 模块B（检查点验证器）
  │     对比 expected_checkpoint_state vs KeyState.label
  │     "模块A说应该进入隐私设置页，Agent 的状态 s_2 标注为隐私设置页" → 达成
  │     "模块A说应该进入密码保险箱页，Agent 的状态序列中没有匹配" → 遗漏
  │     "状态 s_4 标注为 404 错误页" → 执行异常
  │
  ├──▶ 模块C（效率分析器）
  │     状态停留时长 + 状态内步数 = 效率指标
  │     "s_1（设置首页）停留 5 步 12 秒，预期 2 步" → 低效
  │     连续相同的 actionPurpose 在同一状态内 → 重复操作
  │
  ├──▶ 模块D（轨迹差分判定器）
  │     transition_type: "back" 标注 → 回退偏差
  │     transition_type: "loop" 标注 → 循环偏差
  │     预期路径 vs 实际 StateGraph 分支 → 三分类判定
  │
  └──▶ 重复动作 / 规划失效检测器
        StateGraph 内同状态多步 → 规则检测可直接用结构化输入
        替代当前从原始动作序列反推的方式
```

### 5.2 与 _deduped.json 的关系

```
_deduped.json       → 机械层：每步做了什么动作（type/坐标/控件名/耗时）
_stategraph.json    → 语义层：Agent 经过了哪些页面状态（标签/指纹/切换证据）

两者互补：
  _deduped.json 回答 "第3步 Agent 点击了什么"
  _stategraph.json 回答 "第3步属于哪个页面状态、这个状态持续了多久"
```

## 6. 实现路径

### 6.1 分阶段交付（v2.0 简化版）

```
Phase 1 — MVP（1~2 周）
├── clearRes.gzip 解析模块
│   ├── rawPage OCR 树解析
│   └── actionPurpose 提取与对齐
├── actionPurpose 语义分类器（规则匹配）
├── OCR 树指纹计算
├── 控件名称解析（icon → sibling text 关联）
├── pHash 视觉验证
├── StateGraph 构建与输出
└── 集成到 process_gui_end_to_end.py 管线

Phase 2 — 增强（1~2 周）
├── 回退/环路检测
├── 稳定态/过渡态/中间态三分
├── 规则标签生成器
├── 与 convert_to_check_e2e.py 对接（StateGraph → step_level_instruction 增强）
└── 阈值标定（pHash/OCR 指纹相似度）

Phase 3 — 对接判定（1~2 周）
├── StateGraph → 模块B（检查点验证器）输入适配
├── StateGraph → 模块C（效率分析器）输入适配
├── StateGraph → 模块D（轨迹差分判定器）输入适配
├── Darwin VLM 验证 StateGraph 状态准确性
└── 可选 VLM 语义标签增强
```

### 6.2 技术栈（基于已有依赖）

| 组件 | 方案 | 说明 |
|------|------|------|
| OCR 树解析 | 原生 Python json | clearRes.gzip → 解压 → loads |
| pHash | `imagehash` 库 | 已有 |
| OCR 指纹 | SHA-256 + Python stdlib | 零依赖 |
| actionPurpose 分类 | `re` 正则 | 零依赖 |
| StateGraph 存储 | JSON 序列化 | 与现有管线一致 |
| VLM 标签（可选） | Qwen3-VL（已有部署） | Phase 3 按需调用 |

## 7. 风险与缓解

| 风险 | 缓解 |
|------|------|
| `actionPurpose` 表述不规范，规则匹配不准 | 积累失败 case → 补充正则；Phase 2 加入 LLM 分类器兜底 |
| `clearRes` OCR 树对某些 App 质量差（游戏/Canvas） | 降级为纯视觉模式：仅靠 pHash + 步间间距判断状态切换 |
| 状态指纹相同但实际是不同状态（如两个外观相同的列表页） | Phase 2 加入上下文文本差异（列表项 text 不同） |
| `actionPurpose` 不可用（旧数据） | 回退到纯视觉边界检测（v1.0 的级联过滤逻辑） |

## 8. 与 v1.0 的核心差异总结

| 维度 | v1.0 | v2.0 |
|------|------|------|
| **数据假设** | 标准 a11y tree | `clearRes.json` OCR 树（实际可用） |
| **状态边界检测** | 动态滑动窗口 + 四层融合判据 | actionPurpose 语义标注 + pHash/OCR 指纹验证 |
| **语义标签** | CLIP 嵌入 + VLM 生成 | actionPurpose 提取 + clearRes text |
| **元素名称** | a11y node.text | clearRes 树 sibling text 关联 |
| **计算成本** | 每对帧都需 pHash→SSIM→tree edit→CLIP | 仅边界候选点做验证 |
| **MVP 周期** | 7+ 周 | 1~2 周 |
| **与现有管线关系** | 独立模块 | 嵌入 process_gui_end_to_end.py，产出 _stategraph.json |
| **下游对接** | 通用状态图 | 精确对接 Intent-vs-Execution 双层模型 |
