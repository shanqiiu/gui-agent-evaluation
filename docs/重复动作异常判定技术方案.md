# Repeated-Action Detection Plan

Version: v1.0
Updated: 2026-07-16
Status: topic design aligned with the new baseline

## 1. Goal

Detect repeated GUI operations in an actual agent trajectory without app/domain hardcoding.

A repeated action is an equivalent action performed against an equivalent target or state without meaningful page, OCR, visual, or checkpoint progress.

## 2. Current Implementation

The current baseline is implemented by:

- `src.common.repeated_action_detector`
- `src.evaluator.repeated_baseline`
- `src.evaluator.state_evidence`

Inputs:

- hydrated `payload`
- `ab_report`
- `verification_report`
- `state_sequence`

Output:

- `repeated_prediction.json`
- `baseline_result.json.repeated_prediction`

## 3. Evidence Used

| Evidence | Purpose |
|---|---|
| action type/text/coordinates | action equivalence |
| AB label and page descriptions | whether a step caused expected progress |
| state transitions | whether page/state changed |
| checkpoint verification | whether task progress increased |
| visual evidence | pHash, SSIM, pixel diff, changed regions |
| OCR/rawPage | text/state change evidence |

Non-GUI actions are ignored: user reply, broadcast/speech/TTS, assistant/system/reply/speak actions.

## 4. Detection Principle

```text
equivalent action
+ equivalent target or same state
+ no state transition
+ no checkpoint progress
+ not an allowed repeated pattern
=> repeated_action
```

The detector must not declare a high-confidence repeated abnormality using action text alone.

## 5. Reasonable Repetition

Do not mark these as abnormal unless evidence shows no progress:

- scrolling through new list content
- retry after explicit loading/network error
- closing different popup instances
- multi-select over different coordinates/items
- text editing where the field content changes
- do-nothing/wait when the page is still loading

## 6. Current Output Shape

```json
{
  "label": "normal|abnormal",
  "type": "repeated_action",
  "severity": "none|low|medium|high",
  "confidence": 0.0,
  "ranges": [],
  "summary": "...",
  "metrics": {
    "action_count": 0,
    "repeated_range_count": 0
  }
}
```

## 7. Known Limits

- Target equivalence still relies partly on action text and coordinates.
- The detector benefits strongly from good `state_sequence` and checkpoint verification.
- OCR and visual thresholds still need calibration on a labeled set.

## 8. Next Work

1. Add labeled repeated-action cases.
2. Report false positives for reasonable scroll/retry.
3. Calibrate state boundary thresholds.
4. Add target-region evidence when screenshot crops are available.
