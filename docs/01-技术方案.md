# GUI Agent 执行轨迹自动评估技术方案

> 版本：v3.0
> 更新日期：2026-07-15
> 状态：当前唯一规范性技术方案（Source of Truth）

## 1. 方案结论

本项目采用“结构化证据 + 多模态验证 + 可配置规则 + 标定后聚合”的混合评估架构，评估 GUI Agent 的任务结果、执行过程、异常行为、恢复能力和证据质量。

当前开发重点不是继续增加独立判定器，而是建立统一数据契约、真实信号、端到端编排和标注评测闭环。

规则方法的定位调整为：

```text
确定性规则：可以直接形成结论
高精度规则：形成强证据，参与聚合
启发式规则：只形成候选异常，不独立定罪
VLM/LLM：负责页面语义、检查点达成和因果关系等非确定性判断
```

在完成标注集校准前，不使用未经验证的固定权重输出单一总分，不将“无证据”解释为“正常”，也不将 Agent 的自述作为真值。

## 2. 目标与边界

### 2.1 核心目标

1. 对 `utg.json`、`clearRes`、截图和 Darwin 结果进行统一解析。
2. 形成稳定、可版本化的步骤、页面、检查点和异常证据模型。
3. 输出任务完成度、执行异常、规划异常、效率和恢复能力等独立指标。
4. 支持全量离线评估、级联评估和纯文本快速预筛。
5. 使用人工标注集量化 Precision、Recall、F1、首错定位误差和置信度校准质量。

### 2.2 非目标

- 不依赖唯一标准操作轨迹；同一任务允许存在多条合法路径。
- 不把 actionPurpose、Agent thought 或 finished 声明视为完成真值。
- 不使用截图文件名、路径哈希或进程不稳定哈希代替真实视觉/结构特征。
- 不在缺少页面或检查点证据时强行输出正常/异常二分类。
- 不在未标定前承诺固定的综合分权重或“准确率提升 30%”。

## 3. 设计原则

### 3.1 证据优先

每个结论必须引用可追溯证据：步骤索引、动作、控件、前后页面、检查点、模型输出或规则命中条件。

### 3.2 信号分级

| 等级 | 信号 | 示例 | 使用方式 |
|---|---|---|---|
| S0 确定性 | 结构化协议事实 | finished 后仍有真实动作、非法坐标、显式失败码 | 可直接判定 |
| S1 客观观测 | rawPage/OCR、真实图像特征、系统事件 | 控件命中、页面文本、加载状态、截图变化 | 强证据 |
| S2 模型判断 | Darwin AB、VLM 检查点验证 | 页面是否跳转、目标状态是否达成 | 带置信度证据 |
| S3 Agent 自述 | actionPurpose、thought、planning text | Agent 声称要点击某控件 | 仅用于交叉验证 |

当信号冲突时，优先级通常为 `S0 > S1 > S2 > S3`。S2 与 S1 明显冲突时输出 `uncertain` 并进入复核，而不是静默覆盖。

### 3.3 允许未知

所有主要判定至少支持：

```text
normal / abnormal / uncertain / not_applicable
```

缺少截图、检查点、页面状态或模型调用失败时，应降低覆盖率并输出原因。

### 3.4 可复现

- 图像指纹必须读取图像内容。
- OCR 指纹必须使用稳定序列化和稳定哈希。
- 规则、Prompt、模型、阈值和 App 配置均记录版本。
- 相同输入和相同配置应生成相同结构化结果。

## 4. 总体架构

```text
原始任务数据
  utg.json + clearRes + screenshots + runtime events
       │
       ▼
L0 统一预处理与契约校验
  NormalizedTask / NormalizedStep / ImageReference
       │
       ├───────────────┐
       ▼               ▼
L1 任务检查点       L1 客观证据提取
  LLM + RAG          rawPage/OCR/图像/动作/时序
  CheckpointSpec     StepEvidence / StateEvidence
       │               │
       └───────┬───────┘
               ▼
L2 分层验证
  AB 动作效果验证 / 检查点验证 / 硬规则 / 异常候选规则
               │
               ▼
L3 轨迹分析
  进度对齐 / 重复与循环 / 终止检查 / 恢复验证 / 首错候选
               │
               ▼
L4 综合评估器
  证据冲突处理 / 置信度聚合 / 模块化指标 / 结构化报告
               │
               ▼
L5 标注评测与校准
  Precision / Recall / F1 / Calibration / Regression
```

## 5. 统一数据契约

### 5.1 EvaluationInput

```python
@dataclass
class EvaluationInput:
    schema_version: str
    task_uuid: str
    instruction: str
    app_id: str
    app_version: str
    steps: list[NormalizedStep]
    checkpoints: list[CheckpointSpec]
    image_mode: str                 # path | base64 | object_uri
    image_base_dir: str
    metadata: dict
```

要求：

- `task_uuid` 必须进入 payload 和最终报告。
- 图片引用必须显式声明模式，只在统一的 ImageResolver 中转换一次。
- 步骤索引使用源索引和归一化索引两个字段，禁止混用。
- finished 是终止声明，不计入真实动作数。
- 所有外部输入在进入判定器前完成 schema 校验。

### 5.2 CheckpointSpec

```python
@dataclass
class CheckpointSpec:
    checkpoint_id: str
    name: str
    required: bool
    expected_state: str
    preconditions: list[str]
    completion_evidence: list[str]
    forbidden_evidence: list[str]
    source: str                     # explicit | rag_template | llm
    confidence: float
```

LLM 输出不能直接进入下游。必须完成字段校验、ID 补全、去重和必要检查点约束检查。

### 5.3 StepEvidence

```python
@dataclass
class StepEvidence:
    source_step_index: int
    normalized_step_index: int
    action_type: str
    action_target: str
    action_coordinates: list[float]
    action_purpose: str
    control_match: dict
    page_before: dict
    page_after: dict
    visual_change: dict
    ab_result: dict
    evidence_quality: str           # complete | partial | missing
```

### 5.4 RuleFinding

```python
@dataclass
class RuleFinding:
    rule_id: str
    rule_version: str
    label: str                      # abnormal | normal | uncertain
    severity: str
    confidence: float
    step_range: tuple[int, int]
    evidence: list[str]
    missing_signals: list[str]
```

规则输出统一为 Finding，不允许各模块维护不同的 `label/ranges/events` 方言。

## 6. 数据预处理与 App 知识

### 6.1 统一预处理

保留现有 `src/preprocessor` 的一次解析、多份产出模式，但产物职责调整为：

| 产物 | 定位 |
|---|---|
| `payload.json` | 标准评估输入，不再只服务 Darwin |
| `_deduped.json` | 人类审计和问题排查摘要 |
| `_stategraph.json` | 可选的状态证据产物，不作为默认真值 |
| `manifest.json` | schema、文件哈希、App、模型和配置版本 |

### 6.2 App 识别

不能将 App 固定为 `settings`。App 解析顺序：

1. 使用源数据中的 bundle/package name。
2. 使用 CurrentPageUrl、WindowName 等上下文。
3. 无法识别时使用 `unknown`，RAG 不加 App 过滤。
4. 报告中记录 App 识别置信度。

### 6.3 RAG 知识库

- 文档按 App、版本、页面和能力进行 metadata 管理。
- 摄入使用 upsert，支持文档覆盖和删除。
- 知识命中只辅助生成检查点，不作为检查点达成证据。
- 未命中知识库时允许纯 LLM 分解，但降低 checkpoint source confidence。

## 7. 状态与页面证据

状态图保留为辅助抽象，但必须重建其信号基础。

### 7.1 边界候选

以下事件只生成候选边界：

- open、back、导航点击、提交、确认、clarify、finished。
- actionPurpose 声称进入/返回某页面。
- OCR 页面结构显著变化。
- 真实图像 pHash/SSIM 显著变化。
- BundleName、PageUrl、WindowName 变化。

### 7.2 边界确认

普通边界至少需要两个相互独立信号；显式 App/PageUrl 切换等 S0/S1 强信号可以单独确认。

```text
actionPurpose + 路径文件名变化               => 不确认
actionPurpose + OCR 结构变化                  => 可确认
真实图像变化 + OCR 结构变化                   => 可确认
PageUrl 变化                                 => 强确认
仅视觉变化且存在动画/键盘/加载                => uncertain
```

### 7.3 稳定指纹

- 图像：从像素计算 pHash/dHash，可选 SSIM。
- OCR：规范化节点类型、文本、bounds 后使用 SHA-256。
- 禁止使用 Python `hash()`。
- 指纹相同只代表候选同状态，不直接代表回退。
- 同页滚动需要结合可视文本窗口和动作上下文识别。

### 7.4 回退与循环

回退需要验证：

1. 当前状态与历史状态相似。
2. 中间确实访问了不同状态。
3. 存在 back、返回控件或导航语义证据。

循环需要验证状态序列和动作序列共同重复，例如 `A-click-B-back-A` 重复两次。单一 OCR 指纹重复不能判定循环。

## 8. 检查点生成与验证

### 8.1 生成策略

检查点来源优先级：

1. 任务或测试用例显式提供的检查点。
2. App 模板/RAG 中的版本化必要步骤。
3. LLM 自动分解。

每个检查点必须描述“可观察完成条件”，避免仅输出动作名称：

```text
不推荐：点击隐私和安全
推荐：隐私和安全页面已打开，标题或页面核心控件可见
```

### 8.2 轨迹对齐

禁止按检查点数量将其均匀分配到步骤。

采用候选检索 + 单调对齐：

1. 使用动作目标、页面 OCR、actionPurpose 和状态标签召回候选步骤。
2. 对每个检查点在候选步骤的 after 状态上执行验证。
3. 通过动态规划或单调匹配保持检查点顺序，但允许可选步骤和合法捷径。
4. 无可靠候选时输出未对齐，而不是随机绑定步骤。

### 8.3 分层验证

```text
Tier 0：结构规则
  显式状态码、PageUrl、控件属性、输入框值

Tier 1：OCR/rawPage
  标题、关键文本、控件状态、必要字段

Tier 2：VLM
  页面语义、跨模态状态、遮挡、动态结果

Tier 3：人工复核
  核心检查点低置信度或信号冲突
```

关键检查点不允许只使用动作文本匹配判定达成。

## 9. 规则判定框架

### 9.1 Rule 接口

```python
class EvaluationRule(Protocol):
    rule_id: str
    version: str

    def applies(self, context: EvaluationContext) -> bool: ...
    def evaluate(self, context: EvaluationContext) -> list[RuleFinding]: ...
```

规则只读取统一 FeatureStore，不自行重复解析 payload 或 Darwin 方言。

### 9.2 规则分级

| 类型 | 示例 | 是否可独立形成最终异常 |
|---|---|---|
| 硬规则 | finished 后继续操作、显式崩溃码、坐标越界 | 是 |
| 高精度规则 | 同控件连续点击且页面客观无变化 | 满足完整证据时可以 |
| 启发式规则 | 连续滑动过多、返回后疑似恢复、动作文本相似 | 否 |
| 因果规则 | 首错导致后续级联 | 否，需要模型或标注支持 |

### 9.3 重复动作

重复异常需要满足：

```text
动作等效
+ 目标等效
+ 页面/检查点无进展
+ 不属于合理重试、滚动、连续编辑或多选白名单
```

在没有页面和检查点信号的纯文本模式中，只输出 `suspected_repeat`，不能输出确定性重复异常。

### 9.4 规划失效

规划失效拆为独立证据：

- 必要检查点缺失。
- 前置条件违反。
- 提前终止。
- 完成后继续执行。
- 目标或计划疑似不一致。

“目标误解”和“步骤顺序错误”必须结合检查点、页面状态和任务语义。字符串相似度只能用于候选召回。

### 9.5 效率

效率只在任务完成或可比较的检查点覆盖范围内计算。核心指标包括：

- 无进展动作率。
- 重复动作步数。
- 完成关键检查点的有效步骤数。
- 已验证循环/回退次数。
- 各状态停留时间。

连续滑动次数、总步骤数等只能作为上下文特征，不直接代表低效。

### 9.6 偏差与恢复

保留以下报告标签：

- `no_material_deviation`：存在路径差异，但结果和核心检查点正确。
- `verified_recovery`：存在已验证错误，后续状态明确恢复。
- `unrecovered_failure`：错误或缺失检查点最终未恢复。
- `suspected_cascade`：存在首错和后续关联证据，但因果尚未完全确认。
- `uncertain`：证据不足。

在没有因果证据时，不直接使用 `cascading` 作为确定性结论。

## 10. 综合评估器

综合评估器是唯一对外报告入口，负责：

1. 调用或接收各模块 Finding。
2. 合并同一异常的重复证据。
3. 处理规则、OCR、VLM 和 Agent 自述冲突。
4. 计算覆盖率和证据质量。
5. 输出模块化结果和最终任务状态。

最终任务状态：

```text
pass / fail / partial / uncertain / not_evaluable
```

输出至少包含：

- `schema_version`、`task_uuid` 和版本信息。
- `task_status`、`confidence`、`evidence_coverage`。
- 检查点统计和逐检查点证据。
- 标准化 Finding 列表。
- 模块级指标和缺失信号。

在完成标定前，报告模块化指标，不输出缺乏统计依据的 0-100 总分。

## 11. 运行模式

### 11.1 完整模式

使用 rawPage/OCR、真实图像、Darwin AB、检查点 VLM 和规则引擎。用于正式离线评估和标注集验证。

### 11.2 级联模式

1. 硬规则和文本规则全量运行。
2. OCR/rawPage 验证边界候选。
3. 仅对核心检查点、导航/提交/终端和冲突样本调用 VLM。
4. 低置信度样本进入人工复核或完整模式。

### 11.3 纯文本预筛

只使用动作类型、坐标、文本、方向和显式终止信息，适用于批量预筛或 VLM 不可用场景。

允许输出硬协议异常、疑似重复动作、finished 后继续操作和明显动作序列错误。

不允许输出页面跳转正确、检查点真实达成、整体意图完成、目标误解或级联因果。

## 12. 标注集与评测

### 12.1 最小标注集

第一阶段建议至少 300 条任务：

| 类别 | 建议数量 |
|---|---:|
| 正常且路径不同 | 60 |
| 重复点击/输入/滑动 | 50 |
| 合理滚动/重试/多选 | 40 |
| 必要检查点缺失 | 40 |
| 提前终止/未能终止 | 40 |
| 执行错误或页面无响应 | 30 |
| 已验证恢复 | 20 |
| 环境异常/功能 bug | 20 |

标注内容包括任务状态、检查点状态、异常类型、异常区间、首错候选、是否恢复和证据充分性。

### 12.2 指标

| 模块 | 主要指标 |
|---|---|
| 检查点验证 | Macro F1、核心检查点 Recall、uncertain rate |
| 重复动作 | Precision、Recall、合理滚动误杀率 |
| 规划失效 | Macro F1、功能 bug 误归因率 |
| 首错定位 | 平均绝对步骤误差、Top-2 命中率 |
| 综合结果 | Task pass/fail F1、Brier score、ECE |

首阶段目标：

- 重复动作 Precision >= 0.85。
- 规划失效 Precision >= 0.85、Recall >= 0.70。
- 核心检查点 Recall >= 0.90。
- 合理滚动/重试误杀率 <= 0.10。
- 首错定位平均误差 <= 2 steps。

### 12.3 回归机制

- 每条新增规则必须附带正例、反例和边界例。
- 阈值调整必须输出旧/新混淆矩阵。
- App 配置变更必须跑对应 App 回归集。
- 不以 mock 测试替代真实图片和真实轨迹评测。

## 13. 模块迁移决策

| 当前模块 | 决策 | 主要工作 |
|---|---|---|
| `src/preprocessor` | 保留并增强 | schema、task_uuid、图片模式、manifest |
| `src/decomposer` | 保留并重构 | App 自动识别、输出校验、RAG upsert |
| `src/oracle` | 作为 Darwin Adapter | 隔离旧字段和服务实现 |
| `src/state_extractor` | 重构 | 真实 pHash、稳定 OCR 指纹、可靠回退/循环 |
| `src/verifier` | 重构 | 候选检索与单调对齐、统一图片解析 |
| `src/common` | 设为唯一规则实现 | 合并 `src/oracle` 重复检测器 |
| `src/efficiency` | 保留为证据模块 | 删除未经标定的强结论 |
| `src/trajectory` | 降级为偏差证据聚合 | 不直接声称因果级联 |
| `src/evaluator` | 新建 | 统一编排、冲突处理和报告 |

## 14. 实施路线

### Phase 0：契约与工程基线

- 建立 schema 和统一枚举。
- 补齐 task_uuid、图片模式和步骤索引。
- 修复测试收集、依赖、启动方式和文档。
- 确定 `src/common` 为唯一规则实现。

### Phase 1：可信证据层

- 实现真实图像 pHash/SSIM。
- 实现稳定 OCR 指纹。
- 增加 ImageResolver 和 AppResolver。
- 建立状态边界真实样本测试。

### Phase 2：端到端评估基线

- 实现 CheckpointAligner。
- 将 Verifier、规则、效率和轨迹模块接入 Evaluator。
- 提供统一 CLI/API 和 EvaluationReport。
- 支持完整、级联和纯文本预筛模式。

### Phase 3：标注与标定

- 建立 300 条最小标注集。
- 标定规则阈值、VLM Prompt 和置信度。
- 输出模块级 Precision/Recall/F1。
- 建立持续回归集。

### Phase 4：模型增强

仅在规则和通用 VLM 达到瓶颈后考虑专用轨迹审计小模型、学习型检查点对齐器、级联因果分类器和跨 App/版本自适应阈值。
