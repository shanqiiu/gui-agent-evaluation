# GUI Agent 执行轨迹自动评估技术方案

版本：v4.1
更新日期：2026-07-17
状态：当前唯一规范性技术方案

## 1. 当前方向

项目当前采用 `src.evaluator`、`src.common` 和 `src.verifier` 下的独立新基线。`src/oracle` 仅作为 legacy 参考，不属于默认执行链路。

当前流程：

```text
payload.json
  -> hydrate_payload_images
  -> ABValidator
  -> build_state_sequence
  -> match_checkpoint_intents
  -> align_checkpoints_to_steps
  -> CheckpointVerifier
  -> detect_repeated_actions
  -> detect_planning_failure
  -> baseline_result.json
```

关键架构选择是两阶段 checkpoint 判定：

1. 意图召回：把 checkpoint 映射到可能的实际状态或步骤。
2. 执行验证：只对召回候选使用截图和 VLM 证据验证。

## 2. 设计原则

- 不在通用模块中写 App 或业务场景硬编码。
- 不把 `agent_purpose` 当作真值。
- 不按步骤数量均匀分配 checkpoint。
- 除 mock 测试外，checkpoint 验证必须使用真实截图或真实状态证据。
- 不把缺失证据解释为正常。
- action text、page description、state evidence、intent recall、verifier output 必须使用统一 source step index。

## 3. 证据分层

| 层级 | 证据 | 当前承载 |
|---|---|---|
| S0 | 结构事实：动作类型、源步骤索引、finished | `payload.seq_info` |
| S1 | 客观观测：图片、OCR/rawPage、pHash、SSIM、pixel diff | `state_sequence` |
| S2 | 模型判断：AB 页面描述、checkpoint VLM 结果、可选 LLM 重排 | `ab_report`、`verification_report`、`intent_matches` |
| S3 | Agent 自述：`agent_purpose`、动作文本 | 只用于意图召回和状态聚合 |

## 4. 预处理契约

`src.preprocessor.pipeline` 为每个任务生成：

```text
payload.json
_deduped.json
_stategraph.json
catchDataTurnId*.jpg
```

基线只消费 `payload.json`。重要字段：

| 字段 | 是否建议 | 含义 |
|---|---|---|
| `instruction` | 必需 | 用户任务 |
| `seq_info` | 必需 | 带源索引的动作和截图序列 |
| `_image_base_dir` | path 模式必需 | 截图基准目录 |
| `_checkpoints` | 建议 | 显式或 LLM+RAG 生成的结构化检查点 |
| `_action_purposes` | 建议 | Agent 实际每步意图 |
| `_ocr_pages` | 建议 | rawPage/OCR 证据 |
| `_interruption_events` | 可选 | 从主执行轨迹剥离的非 GUI 中断事件，如 `clarify` |

图片只在 `hydrate_payload_images` 中统一解析一次。禁止把路径字符串当作图片内容传给 VLM。

`clarify(...)` 不进入 `seq_info`，不能占用可执行动作步骤，也不能消费或错位后续 `action_purpose`。该类事件只写入 `_interruption_events`，由统一异常事件输出映射到“异常中断响应”。

checkpoint 粒度契约：

- 一个 checkpoint 表示一个不可缺少、路径无关的关键可观察子状态。
- checkpoint 不表示点击、输入、滑动等具体操作。
- `name` 是简短子状态名称，`expected_state` 是截图/OCR/页面结构验证契约。
- 一个 checkpoint 只能承载一个状态边界，禁止复合阶段和重复状态。
- Decomposer 在生成后执行质量校验，发现问题时最多自动修正一次。

## 5. 基线编排

`src.evaluator.repeated_baseline.run_repeated_baseline` 执行：

1. 读取 `payload.json`。
2. 将截图路径水合为 Base64。
3. 对相邻步骤运行 `ABValidator`。
4. 基于 AB、视觉和 OCR 证据构建初始 `state_sequence`。
5. 运行 `match_checkpoint_intents`。
6. 运行 `align_checkpoints_to_steps`。
7. 对召回命中的检查点运行 `CheckpointVerifier`。
8. 带验证进度重建 `state_sequence`。
9. 运行 `detect_repeated_actions`。
10. 运行 `detect_planning_failure` 聚合任务级规划失效。
11. 写出所有 JSON 报告。

批量模式会递归查找预处理输出目录下的 `payload.json`。

## 6. 意图召回

实现位置：`src.verifier.alignment.match_checkpoint_intents`。

输入信号：

- checkpoint name、preconditions、expected_state
- 实际动作文本
- 实际 `agent_purpose`
- AB 页面描述
- 聚合状态描述
- 通用动作类型兼容性
- 可选 LLM 重排

输出文件：`intent_matches.json`。

关键状态：

| 状态 | 含义 |
|---|---|
| `matched=true` | 找到候选步骤或候选状态 |
| `matched=false`, `confidence=unmatched_intent` | 没有可靠实际意图候选 |

`unmatched_intent` 会阻止该 checkpoint 进入执行验证。

## 7. 执行验证

实现位置：`src.verifier.verifier.CheckpointVerifier`。

只验证已经对齐的候选。意图未召回时，image context 会保留：

```json
{
  "step_index": -1,
  "image_available": false,
  "alignment": {"confidence": "unmatched_intent"}
}
```

VLM 验证需要真实截图。mock 模式只用于工程回归。

## 8. 状态证据

实现位置：`src.evaluator.state_evidence` 和 `src.evaluator.visual_evidence`。

状态段聚合：

- source step index
- 页面描述
- 动作类型
- action purpose
- 视觉变化摘要
- OCR/rawPage 指纹和相似度
- evidence quality

该层同时服务重复操作判定和 checkpoint 意图召回。

## 9. 重复操作基线

重复操作检测读取：

- hydrated payload
- AB report
- verification report
- state sequence

用户回复、播报、TTS、assistant/system/reply/speak 等非 GUI 动作会被忽略。

高置信重复异常必须同时具备动作相似和状态/checkpoint 无进展证据。仅文本相似不足以判定高置信异常。

## 10. LLM/VLM 配置

`repeated_baseline` 会自动加载 `.env`。

| 变量 | 用途 |
|---|---|
| `VLM_MODEL_URL`, `VLM_MODEL_NAME`, `VLM_API_KEY` | ABValidator 和 CheckpointVerifier |
| `LLM_MODEL_URL`, `LLM_MODEL_NAME`, `LLM_API_KEY` | Decomposer 和可选意图重排 |

如果 `LLM_*` 缺失，意图重排会回退复用 `VLM_*`。这适配当前远程服务器上一套 OpenAI-compatible 服务同时承担 LLM/VLM 的部署方式。

## 11. 输出契约

`baseline_result.json` 包含：

| 字段 | 含义 |
|---|---|
| `schema_version` | 基线 schema 版本 |
| `task_uuid` | 任务 ID |
| `image_resolution` | 截图水合统计 |
| `ab_report` | AB 验证报告 |
| `intent_matches` | 意图召回输出 |
| `checkpoint_alignments` | 执行候选对齐 |
| `verification_report` | checkpoint VLM 验证 |
| `state_sequence` | 状态证据输出 |
| `repeated_prediction` | 重复操作结果 |
| `planning_failure_prediction` | 规划失效聚合结果 |
| `anomaly_events` | 按 1.2 顶层 taxonomy 投影的统一异常事件（首版承接 repeated/planning/clarify/state interruption） |

## 12. 异常 taxonomy 规范

异常 case 的顶层分类必须与 `docs/GUI_Agent_异常Case_技术洞察.md` 1.2 “异常 case 的定义边界”保持一致：

1. 循环/死循环
2. 重复动作
3. Grounding 错误
4. 规划失效
5. Hallucination
6. 异常中断响应
7. 提前终止

当前 `repeated_prediction` 和 `planning_failure_prediction` 是第一版兼容输出，不等价于完整 taxonomy。统一事件输出使用 `anomaly_events`；首版已承接 repeated ranges、planning failure events、`clarify` 中断事件，以及基于 `state_sequence` 的验证码/登录/权限/Crash/网络加载关键词扫描。后续扩展时，应将执行阻塞、恢复失败、错误级联、效率、证据完整度和高风险操作作为 subtype、recovery outcome、impact、report metric 或 risk attribute，不新增为顶层异常分类。

## 13. 规划失效聚合

实现位置：`src.evaluator.planning_failure.detect_planning_failure`。

当前第一版已经接入 `src.evaluator.repeated_baseline`，单跑会额外写出 `planning_failure_result.json`，批跑汇总会增加：

- `planning_failure_label`
- `planning_failure_subtype`
- `planning_failure_confidence`

当前聚合子类型：

- `missing_required_checkpoint`：必要 checkpoint 在意图层未召回。
- `execution_blocked`：意图匹配成功，但 verifier 判定未达成。
- `premature_termination`：required checkpoint 未完成前出现 finished/done。
- `fail_to_terminate`：required checkpoint 已完成后仍继续执行多个非终止动作。

缺少 `_checkpoints` 或 verifier 证据不确定时输出 `uncertain`，不直接判为规划失败。
## 14. 验证计划

必要评估：

1. 用同一人工标注集分别跑新基线和旧 oracle 链。
2. 先衡量 intent recall rate。
3. 只在召回候选上衡量 checkpoint verification precision/recall。
4. 单独衡量重复操作 precision/recall。
5. 统计缺证据类别：图片缺失、OCR 缺失、意图未召回、VLM 不确定。

## 15. Legacy 边界

`src/oracle` 可用于对比、prompt 参考和旧 API 兼容，但新基线不应 import 它。新代码应使用：

- `src.common.ABValidator`
- `src.evaluator.state_evidence`
- `src.verifier.alignment`
- `src.verifier.verifier`
- `src.common.repeated_action_detector`
