# GUI Agent Evaluation Technical Plan

Version: v4.0
Updated: 2026-07-16
Status: source of truth

## 1. Current Direction

The project now uses an independent baseline implemented under `src.evaluator`, `src.common`, and `src.verifier`. `src/oracle` is legacy reference only and is not part of the default execution chain.

Current pipeline:

```text
payload.json
  -> hydrate_payload_images
  -> ABValidator
  -> build_state_sequence
  -> match_checkpoint_intents
  -> align_checkpoints_to_steps
  -> CheckpointVerifier
  -> detect_repeated_actions
  -> baseline_result.json
```

The key architectural choice is two-stage checkpoint judgment:

1. Intent recall maps checkpoints to likely actual states/steps.
2. Execution verification validates only recalled candidates using screenshots and VLM evidence.

## 2. Design Principles

- Do not use app/domain hardcoding in generic modules.
- Do not treat `agent_purpose` as ground truth.
- Do not evenly distribute checkpoints over steps.
- Do not verify checkpoints without real screenshot or state evidence unless explicitly running mock tests.
- Do not infer normality from missing evidence.
- Preserve source step indexes across action text, page description, state evidence, intent recall, and verifier output.

## 3. Evidence Levels

| Level | Evidence | Current carrier |
|---|---|---|
| S0 | Structured facts: action type, source step index, finished | `payload.seq_info` |
| S1 | Objective observations: image, OCR/rawPage, pHash, SSIM, pixel diff | `state_sequence` |
| S2 | Model judgments: AB page description, checkpoint VLM result, optional LLM rerank | `ab_report`, `verification_report`, `intent_matches` |
| S3 | Agent self-report: `agent_purpose`, action text | intent recall and state aggregation only |

## 4. Preprocessing Contract

`src.preprocessor.pipeline` produces one task directory with:

```text
payload.json
_deduped.json
_stategraph.json
catchDataTurnId*.jpg
```

The baseline consumes `payload.json`. Important payload fields:

| Field | Required | Meaning |
|---|---:|---|
| `instruction` | yes | user task |
| `seq_info` | yes | source-indexed action and screenshot sequence |
| `_image_base_dir` | yes for path mode | screenshot base directory |
| `_checkpoints` | recommended | structured checkpoints from LLM + RAG or explicit source |
| `_action_purposes` | recommended | agent actual intent per step |
| `_ocr_pages` | recommended | rawPage/OCR evidence |

Images are resolved once by `hydrate_payload_images`; paths must not be sent to VLM as if they were image content.

## 5. Baseline Orchestration

`src.evaluator.repeated_baseline.run_repeated_baseline` executes:

1. Load `payload.json`.
2. Hydrate screenshots to Base64.
3. Run `ABValidator` on adjacent steps.
4. Build initial `state_sequence` from AB + visual/OCR evidence.
5. Run `match_checkpoint_intents`.
6. Run `align_checkpoints_to_steps` using intent matches.
7. Run `CheckpointVerifier` for matched checkpoints.
8. Rebuild `state_sequence` with verification progress.
9. Run `detect_repeated_actions`.
10. Write all JSON reports.

Batch mode recursively finds `payload.json` under a preprocess output directory.

## 6. Intent Recall

Implemented by `src.verifier.alignment.match_checkpoint_intents`.

Inputs:

- checkpoint name, preconditions, expected state
- actual action text
- actual `agent_purpose`
- AB page descriptions
- aggregated state descriptions
- generic action type compatibility
- optional LLM rerank

Output file: `intent_matches.json`.

Important statuses:

| Status | Meaning |
|---|---|
| `matched=true` | candidate step/state found |
| `matched=false`, `confidence=unmatched_intent` | no reliable actual intent candidate |

`unmatched_intent` blocks execution verification for that checkpoint.

## 7. Execution Verification

Implemented by `src.verifier.verifier.CheckpointVerifier`.

It verifies only aligned candidates. For unmatched intent:

```json
{
  "step_index": -1,
  "image_available": false,
  "alignment": {"confidence": "unmatched_intent"}
}
```

VLM verification requires real screenshots. Mock mode is only for engineering tests.

## 8. State Evidence

Implemented by `src.evaluator.state_evidence` and `src.evaluator.visual_evidence`.

State segments aggregate:

- source step indexes
- page description
- action types
- action purposes
- visual change summary
- OCR/rawPage fingerprint and similarity
- evidence quality

This layer serves both repeated-action detection and checkpoint intent recall.

## 9. Repeated-Action Baseline

Repeated-action detection reads:

- hydrated payload
- AB report
- verification report
- state sequence

It ignores non-GUI actions such as user reply, speech, TTS, assistant/system replies, and broadcast-only actions.

A repeated-action result requires action similarity plus lack of state/checkpoint progress. Text-only similarity is insufficient for a high-confidence abnormal result.

## 10. LLM and VLM Configuration

`repeated_baseline` auto-loads `.env`.

| Variable | Used by |
|---|---|
| `VLM_MODEL_URL`, `VLM_MODEL_NAME`, `VLM_API_KEY` | ABValidator and CheckpointVerifier |
| `LLM_MODEL_URL`, `LLM_MODEL_NAME`, `LLM_API_KEY` | Decomposer and optional intent rerank |

If `LLM_*` is absent, intent rerank falls back to `VLM_*`. This matches the current remote-server deployment where one OpenAI-compatible endpoint can serve both roles.

## 11. Output Contract

`baseline_result.json` contains:

| Key | Meaning |
|---|---|
| `schema_version` | baseline schema version |
| `task_uuid` | task id |
| `image_resolution` | screenshot hydration stats |
| `ab_report` | AB validation report |
| `intent_matches` | intent recall output |
| `checkpoint_alignments` | execution candidate alignment |
| `verification_report` | checkpoint VLM verification |
| `state_sequence` | state evidence output |
| `repeated_prediction` | repeated-action result |

## 12. Planning-Failure Direction

Planning failure is not yet a complete independent module in the new chain. The next implementation should aggregate:

- missing required checkpoints
- unmatched intent
- verifier uncertain/failure
- premature finished
- completion after which actions continue
- repeated loops that block checkpoint progress

This should be implemented after collecting enough `intent_matches` and `verification_report` samples.

## 13. Validation Plan

Required evaluation:

1. Run new baseline on the same manually labeled set as the old oracle chain.
2. Measure intent recall rate first.
3. Measure checkpoint verification precision/recall only for recalled candidates.
4. Measure repeated-action precision/recall separately.
5. Track missing evidence categories: image missing, OCR missing, intent unmatched, VLM uncertain.

## 14. Legacy Boundary

`src/oracle` remains useful for comparison, prompt references, and old API compatibility. It should not be imported into the new baseline. New code must use:

- `src.common.ABValidator`
- `src.evaluator.state_evidence`
- `src.verifier.alignment`
- `src.verifier.verifier`
- `src.common.repeated_action_detector`
