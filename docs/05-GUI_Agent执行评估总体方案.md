# GUI Agent 执行评估总体方案

版本：v1.0  
更新日期：2026-07-17  
定位：面向方案介绍、技术交流和评审的代码无关说明

## 1. 方案总述

### 1.1 一句话概括

本方案将一条自然语言任务转化为一组可验证的关键子状态，再把 Agent 的实际操作轨迹聚合成实际状态序列，通过“任务意图匹配 + 真实执行状态检查”判断任务是否完成、执行是否合理，并进一步评估执行效率和用户体验。

### 1.2 需要解决的问题

传统 GUI Agent 评估通常只关注最终页面是否正确，难以回答以下问题：

- Agent 是否真正理解了任务目标。
- Agent 是否完成了任务中不可缺少的关键阶段。
- 某个操作意图正确但实际没有生效时，问题发生在哪里。
- Agent 是否通过大量重复、绕路或试错才得到最终结果。
- 缺少截图或页面证据时，评估结果是否仍然可信。

因此，本方案不仅判断“最终有没有完成”，还回答“通过什么过程完成、过程是否合理、证据是否充分”。

### 1.3 总体架构

```text
任务指令
  + 模型能力
  + RAG 领域知识
        ↓
任务理解与约束分解
        ↓
结构化关键子状态
        ↓
实际操作、页面与视觉变化聚合
        ↓
关键检查点映射与单阶段状态检查
        ↓
任务完成能力与合理性评估
  + 执行效率与体验评估
        ↓
结构化评估结果与证据说明
```

方案遵循三个原则：

1. 评估任务状态，而不是绑定某一种固定操作路径。
2. 先判断 Agent 是否具备对应意图，再验证操作是否真实生效。
3. 证据不足必须输出不确定，不能将缺少证据解释为正常。

## 2. 输入与智能基础

### 2.1 基础输入

评估系统的核心输入包括：

| 输入 | 作用 |
|---|---|
| 任务指令 | 定义用户希望 Agent 完成的目标 |
| 实际操作序列 | 描述 Agent 执行了哪些点击、输入、滑动、返回和终止操作 |
| Agent 操作意图 | 表示 Agent 在每个阶段试图完成什么 |
| 页面截图序列 | 证明操作前后页面是否发生真实变化 |
| 页面文本与结构 | 提供 OCR、控件树、页面元素和状态等可观察证据 |
| 时间与步骤信息 | 支持效率、等待、重复和终止行为分析 |

任务指令是评估目标，实际轨迹是被评估对象，截图和页面结构是主要事实证据，Agent 自述意图只作为辅助信息，不能直接作为任务完成的证明。

### 2.2 模型调用分工

方案使用不同模型完成不同层次的推理，避免让单一模型承担全部判断。

| 能力 | 主要职责 | 不应承担的职责 |
|---|---|---|
| 大语言模型 | 理解任务、分解子状态、处理约束、完成语义匹配 | 不能仅凭文字宣告页面状态已经达成 |
| 视觉语言模型 | 理解操作前后截图、判断关键状态是否真实出现 | 不直接决定完整任务规划是否合理 |
| 客观视觉与结构分析 | 计算页面变化、文本变化、区域变化和状态边界 | 不负责解释复杂任务语义 |

模型调用按任务阶段组织：

1. 任务开始前，语言模型结合 RAG 知识生成关键子状态。
2. 轨迹评估时，语言模型负责预期子状态与实际意图的语义映射。
3. 映射成功后，视觉语言模型检查相应阶段的真实截图和页面状态。
4. 全轨迹完成后，规则与证据聚合层形成任务级和效率级结论。

这种分工让语言模型负责“理解”，视觉语言模型负责“看见”，客观指标负责“测量”，聚合层负责“下结论”。

### 2.3 RAG 知识库构建

RAG 的作用是为任务分解提供领域知识，使模型理解特定 App、平台和业务中的页面关系、功能入口和可观察结果。RAG 不直接输出评估结论，也不替代语言模型。

知识来源可以包括：

- 产品和功能说明文档。
- App 操作手册与标准操作流程。
- 人工测试用例和验收标准。
- 页面名称、控件含义及页面跳转关系。
- 历史成功轨迹和常见异常案例。
- 不同版本、地区和设备下的界面差异说明。

知识库构建过程分为六步：

1. 知识采集：汇总产品文档、测试用例、页面说明和成功经验。
2. 清洗与去重：删除失效、冲突和重复内容，保留版本信息。
3. 语义切分：按功能、页面、任务阶段和可验证状态切分知识，而不是机械按固定字数截断。
4. 元数据标注：记录 App、平台、版本、页面、功能域、语言和有效时间。
5. 向量化索引：建立语义检索索引，同时保留关键词和元数据过滤能力。
6. 检索与重排：根据任务指令召回相关知识，并按相关性、版本和适用范围重排。

RAG 检索后的执行关系为：

```text
任务指令
  -> 检索相关领域知识
  -> 将知识作为任务背景交给语言模型
  -> 语言模型生成结构化关键子状态
```

即使 RAG 成功，语言模型仍然会执行。RAG 提供事实背景，语言模型负责结合具体任务完成推理；如果 RAG 没有命中，系统仍可退化为纯语言模型分解，但需要降低知识可信度标记。

## 3. 任务子状态、约束分解与结构化

### 3.1 任务子状态定义

任务子状态是任务推进过程中不可缺少、能够通过页面证据观察到、且不依赖某一种固定操作路径的阶段性结果。

例如，对于“搜索目标内容、应用筛选条件并进入详情”的任务，合理的子状态可以是：

- 相关搜索结果已经展示。
- 指定筛选条件已经生效。
- 目标详情页面已经打开。

“点击搜索框”“输入关键词”“点击筛选按钮”属于操作动作，不是关键子状态。Agent 可能通过不同入口、快捷方式或语音方式达到相同状态，评估系统不应因为操作路径不同而误判失败。

### 3.2 子状态粒度原则

每个子状态应同时满足以下要求：

- 必要性：对任务目标有实质贡献，不是无关或可忽略步骤。
- 原子性：只描述一个状态边界，不把多个阶段合并为一项。
- 可观察性：能通过截图、OCR、页面结构或明确系统状态验证。
- 路径无关性：描述结果，不限定必须点击哪个坐标或经过哪个固定页面。
- 可排序性：能够说明其前置条件及与其他状态的先后关系。
- 最小充分性：数量足以覆盖任务目标，但不细化为每个微观操作。

通常一个任务包含 1 至 5 个关键子状态；复杂任务可以增加，但应避免把任务计划机械拆成大量动作步骤。

### 3.3 约束分解

任务指令不仅包含目标，也可能包含显式或隐式约束。分解时需要识别：

| 约束类型 | 示例 | 对评估的影响 |
|---|---|---|
| 目标约束 | 必须进入指定详情页 | 决定最终完成状态 |
| 属性约束 | 指定关键词、价格、日期或对象 | 决定状态内容是否正确 |
| 顺序约束 | 先完成 A，再执行 B | 决定子状态依赖关系 |
| 禁止约束 | 不允许发送、不允许提交 | 出现禁用行为即构成异常 |
| 条件约束 | 如果出现弹窗则关闭 | 根据环境是否触发决定是否检查 |
| 终止约束 | 完成后停止操作 | 用于判断提前终止或未及时终止 |

约束分解的目标不是生成更长的计划，而是让每个关键状态具备明确的验证条件和适用范围。

### 3.4 结构化表达

每个关键子状态至少包含以下信息：

| 字段概念 | 含义 |
|---|---|
| 子状态名称 | 对阶段结果的简短描述 |
| 是否必需 | 未达成时是否必然影响任务完成 |
| 前置条件 | 进入该状态前必须满足的条件 |
| 预期状态 | 截图、OCR 或页面结构中应观察到的结果 |
| 顺序与依赖 | 与其他子状态的先后或条件关系 |
| 约束集合 | 该阶段必须满足或不得违反的要求 |
| 知识来源 | 使用了哪些领域知识，以及知识可信度 |

结构化的意义在于把自然语言目标转化为稳定的评估契约，后续可以逐项映射、验证和统计。

## 4. 实际操作的子状态聚合

### 4.1 为什么需要聚合

Agent 的原始轨迹通常是细粒度动作序列。一个关键状态可能由多个连续动作共同完成，例如激活输入区域、输入内容、提交搜索后才形成“搜索结果已展示”状态。因此，不能简单将一个动作对应一个检查点，也不能按步骤数量平均分配检查点。

### 4.2 聚合输入

实际子状态聚合综合使用：

- Agent 每步操作意图。
- 实际动作类型、对象、方向和输入内容。
- 操作前后的页面截图。
- 页面描述、OCR 文本和控件结构。
- 页面区域变化、整体视觉变化和结构变化。
- 已经确认达成的检查点进展。

### 4.3 状态边界识别

当出现以下信号时，可以形成新的实际子状态边界：

- 页面主体内容或页面标题发生明显变化。
- OCR 文本集合或页面结构发生实质变化。
- 页面关键区域出现、消失或状态发生切换。
- 导航、返回、弹窗、结果加载等改变当前交互上下文。
- 某个关键检查点已被真实证据证明达成。

连续滑动、短暂等待或没有产生页面进展的重复点击通常不应单独形成新状态。

### 4.4 聚合结果

聚合后，每个实际子状态包含：

- 覆盖的动作区间。
- 该阶段的 Agent 意图摘要。
- 进入前和完成后的页面状态。
- 视觉、文本和结构变化摘要。
- 状态边界的证据质量。
- 对应的原始步骤索引，便于回溯问题位置。

这一步将“Agent 做了什么”转化为“Agent 实际达到了什么状态”。

## 5. 关键检查点映射与单阶段状态检查

### 5.1 两阶段判定

关键检查点采用“意图层映射在前、执行层验证在后”的方式。

第一阶段是语义意图映射：

- 将任务预期子状态与实际操作意图、实际聚合状态进行语义比较。
- 找出最可能对应的实际动作区间或状态区间。
- 检查顺序是否符合任务依赖。
- 支持一个子状态由多个连续动作完成，也支持一个实际状态同时满足多个检查点。

第二阶段是执行状态验证：

- 只对映射成功的候选阶段检查真实截图和页面证据。
- 判断 Agent 的正确意图是否真正产生了预期结果。
- 意图未匹配时不强行分配截图，也不进行无意义的视觉模型调用。

这种设计能够区分两种问题：

- 没有对应意图：Agent 没有尝试完成必要子任务，属于规划或目标理解问题。
- 有对应意图但状态未达成：Agent 尝试正确，但执行没有生效，属于执行阻塞或环境问题。

### 5.2 单个关键阶段的状态检查

单阶段检查围绕“前置状态、动作过程、结果状态”展开：

```text
前置状态是否满足
        ↓
实际动作区间是否与子状态相关
        ↓
操作后页面是否出现预期状态
        ↓
是否违反任务约束
        ↓
输出达成、未达成或不确定
```

检查内容包括：

1. 前置条件检查：当前页面和历史进展是否允许该阶段发生。
2. 意图一致性检查：实际操作意图是否指向该关键子状态。
3. 页面结果检查：操作后截图是否出现目标页面、目标文本或目标状态。
4. 变化合理性检查：页面变化是否与该阶段预期一致，而不是进入无关页面。
5. 约束检查：属性、顺序、禁止行为和条件要求是否满足。
6. 证据完整性检查：截图、OCR 和结构证据是否足以支持结论。

### 5.3 证据层次

关键阶段结论由多层证据共同支撑：

| 证据层 | 内容 | 作用 |
|---|---|---|
| 结构事实 | 动作、索引、终止标记、输入参数 | 提供可追溯事实 |
| 客观页面证据 | 截图变化、OCR、控件结构、区域变化 | 证明页面是否真实改变 |
| 模型理解 | 页面语义、预期状态匹配、视觉判断 | 解释复杂界面含义 |
| Agent 自述 | 操作意图和规划描述 | 用于召回候选，不能独立证明完成 |

结果分为：

- 达成：意图匹配，且页面证据证明预期状态出现。
- 未达成：存在相关意图，但页面结果与预期不符，或明确违反约束。
- 缺失：必要子状态没有对应的实际意图或状态。
- 不确定：候选存在，但截图、页面结构或模型证据不足。

## 6. 执行效率与体验评估

### 6.1 为什么独立评估效率

任务最终完成不代表执行过程合理。Agent 可能经过大量无效点击、反复返回或长时间等待后偶然完成任务。反之，合理的探索、自我修正和环境恢复不应被简单判为异常。

因此，完成能力与执行效率必须分开评价。

### 6.2 效率评估维度

| 维度 | 关注问题 |
|---|---|
| 有效动作比例 | 每个操作是否带来了页面变化或任务进展 |
| 重复操作 | 是否在同一状态反复执行相同操作且没有进展 |
| 路径冗余 | 是否存在不必要的页面往返和状态循环 |
| 探索开销 | 实际步骤是否显著超过完成任务所需的合理范围 |
| 滑动与等待效率 | 是否出现连续无目标滑动、等待或重试 |
| 终止合理性 | 是否提前结束，或完成后仍继续操作 |
| 恢复能力 | 发生错误后能否识别问题并回到正确路径 |
| 稳定性 | 是否因偶然页面变化、历史记录或弹窗产生持续误判 |

### 6.3 合理重复与异常重复

重复动作不能只按动作文本或坐标相似判定。以下情况可能是合理行为：

- 页面仍在加载，经过合理等待后出现进展。
- 第一次点击未生效，第二次重试后状态改变。
- Agent 发现错误并返回正确页面完成自我修正。
- 长列表中多次滑动，每次都展示了新的内容区域。

只有当“动作重复”与“页面无变化、检查点无进展或状态循环”同时成立时，才应认定为低效或异常重复。

### 6.4 体验评估

体验评估关注用户是否能够接受该执行过程，包括：

- 操作过程是否稳定、连贯和可预测。
- 是否产生明显的无响应、抖动、反复跳转或长时间停顿。
- 是否进行了具有风险或违背用户约束的额外操作。
- 是否在目标完成后及时结束，避免继续改变页面状态。
- 出现不确定情况时是否采取了合理的确认或恢复策略。

## 7. 输出：能力、合理性、效率与体验

### 7.1 任务完成能力和合理性评估

第一类输出回答“任务做成了吗，过程在目标上是否合理”。

主要内容包括：

- 任务总体结论：完成、未完成或不确定。
- 必要检查点完成率。
- 已达成、未达成、缺失和不确定的关键子状态。
- 首个偏离或失败阶段。
- 失败类型：目标理解缺失、规划遗漏、执行未生效、提前终止、未及时终止或约束违反。
- 替代路径合理性：是否使用了不同但有效的执行路径。
- 每项结论对应的截图、页面和步骤证据。

### 7.2 效率和体验评估

第二类输出回答“完成得是否高效、稳定、符合使用体验”。

主要内容包括：

- 总步骤数与有效步骤比例。
- 无进展动作比例。
- 重复、等待、滑动和页面循环区间。
- 相对合理路径的额外开销。
- 自我修正次数及修正是否成功。
- 终止行为是否合理。
- 效率等级和体验等级。
- 影响效率与体验的主要证据。

### 7.3 输出组织

最终报告采用“结论、指标、证据、定位”四层结构：

```text
总体结论
  ├─ 任务完成能力
  ├─ 执行合理性
  ├─ 执行效率
  └─ 用户体验

关键指标
  ├─ 检查点完成率
  ├─ 有效动作比例
  ├─ 冗余与重复比例
  └─ 证据完整度

问题明细
  ├─ 问题类型
  ├─ 首次发生阶段
  ├─ 关联检查点
  └─ 影响范围

证据回溯
  ├─ 实际动作区间
  ├─ 操作前后页面
  ├─ 页面变化摘要
  └─ 判定依据与不确定性
```

综合评分不能掩盖关键失败。例如，一个任务即使操作步骤很少，只要必要检查点未完成，就不能获得较高的任务完成评分；同样，任务完成但存在大量无效操作时，可以获得较高的完成能力评分，但效率和体验评分应明显降低。

## 8. 方案价值总结

本方案的核心不是让模型对整条轨迹直接给出一个主观分数，而是建立一条可以解释和回溯的评估链：

1. 从任务指令中识别目标、约束和关键可观察子状态。
2. 通过 RAG 补充领域知识，由语言模型完成结构化分解。
3. 将实际细粒度操作聚合成真实状态序列。
4. 先进行任务子状态与实际意图、实际状态的语义映射。
5. 再用截图、OCR 和页面结构检查单个关键阶段是否真实达成。
6. 在任务完成判断之外，独立分析重复、绕路、恢复、终止和体验问题。
7. 输出任务能力、执行合理性、效率、体验以及完整证据。

最终形成两条互补的评估主线：

```text
任务能力主线：是否理解目标 -> 是否覆盖关键子状态 -> 是否真实完成 -> 是否合理终止

执行质量主线：是否持续产生进展 -> 是否存在重复或绕路 -> 是否能够恢复 -> 体验是否稳定
```

这种设计既允许 Agent 使用不同但正确的操作路径，也能够准确定位“没有计划、计划正确但执行失败、最终完成但过程低效”等不同问题，为模型能力评估、产品验收、异常分析和后续优化提供统一依据。
