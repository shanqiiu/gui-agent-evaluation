# OKR：端侧执行轨迹自动判定系统

> 版本：v0.1 | 日期：2026-07-06 | 状态：设计阶段

---

## 项目演进脉络

```
O4（前序）                            gui-agent-evaluation（当前）
─────────────────────────────────────────────────────────────
构建异常APP Mock注入                   对执行轨迹做异常自动判定
→ 制造异常场景测                     →  跑完后的轨迹，自动判有没有问题
→ 解决"怎么造异常"                      → 解决"怎么判异常"
```

**输入变化**：以前是构造异常 APP 给 跑，现在是拿到 的执行轨迹（截图序列 + 操作序列），自动判定里面有没有异常、什么类型、严重程度。

**技术基座**： Qwen-VL-7B 微调模型（功能完备性 + 功能可用性判定）+ 五模块评测管线。

## O：构建 端侧执行轨迹自动判定系统，替代人工抽检，实现异常判定自动化与标准化

> 当前 执行异常依赖人工逐条回看轨迹判定，效率低、标准不一。本系统将判定逻辑工程化为自动化管线，输出结构化评测报告，支撑 逻辑执行鲁棒性的量化度量与持续改进。

---

## KR 分解

### KR1：异常检测覆盖 ≥6 类核心异常模式，判定能力对齐学术界 SOTA 分类体系

**衡量标准**：

覆盖 [RoTS](https://arxiv.org/abs/2605.29447)（ICML 2026 Spotlight）11 类 Error Taxonomy 中频率最高的 ≥6 类，同时覆盖 [HalluClear](https://arxiv.org/abs/2604.17284) 的感知层+推理层 Hallucination 检测。

| 异常类型 | 频次排名（RoTS） | 对应模块 | 判定方法 |
|----------|----------------|----------|----------|
| 错误UI元素交互 | #1 (42.9%) | B + D | VLM 截图对比 + 轨迹差分 |
| 无效操作 | #2 (4.5%) | C | Embedding 语义相似度 |
| 遗漏关键步骤 | #3 (4.5%) | B | 检查点达成率 |
| 误解目标 | #5 (1.5%) | B + D | 检查点 + 轨迹差分 |
| 未能终止 | #6 (0.9%) | C | 循环/超时检测 |
| Grounding 错误 | 高频交叉 | B | VLM 截图语义不对齐 |
| 感知层 Hallucination | HalluClear PH.1-4 | 达尔文VLM | 截图状态/元素/属性/关系误判 |
| 推理层 Hallucination | HalluClear RH.1-4 | B + D | 指令违背/上下文不一致/逻辑缺陷/事实捏造 |

**参考基线**：
- 学术界 SOTA：RoTS OSWorld 47.4%，GUI-RobustEval 恢复成功率执行层 40-60%、规划层 <20%
- 本项目基线：达尔文 VLM 判别准确率（需实测标定）

---

### KR2：异常判定准确率 ≥85%，致命级异常召回率 ≥95%

**衡量标准**：在 ≥200 条人工标注轨迹测试集上，综合判定准确率 ≥85%；致命级异常（级联偏差 / 死循环 / 功能不可用）召回率 ≥95%。

**三分类判定体系**：

| 判定维度 | 定义 | 判定来源 | 优先级 |
|----------|------|----------|--------|
| **功能完备性** | 是否走完任务、关键检查点是否达成 | 模块B 检查点达成率 | 基础 |
| **功能可用性** | 走完了但结果对不对 | 达尔文VLM 微调模型 | 核心 |
| **过程异常分类** | 无影响偏差 / 补救性偏差 / 级联偏差 | 模块D 轨迹差分 | 加分 |

**致命级异常（优先保证召回 ≥95%）**：
- 级联偏差（Cascading Failure）→ 小错误被持续放大致任务失败
- 死循环（Search Loop）→ 在有限页面间反复跳转无法突破
- 功能不可用 → 达尔文 VLM 判定不通过

**参考指标**：
- Completion-Correctness Gap：SOTA Agent 完成率 96.0% 但正确率远低于此（F1 = 0.4529）
- 本项目设计天然避免此问题：不依赖 Agent 自报完成，客观判定检查点达成

---

### KR3：单条轨迹端到端判定耗时 ≤30 秒，支撑日均 ≥500 条轨迹处理吞吐

**衡量标准**：典型任务（5-8 步操作、≤10 个检查点）的完整判定管线总耗时 ≤30s（不含模型推理排队）。

**性能分解**：

| 阶段 | 预估耗时 | 占比 | 优化策略 |
|------|---------|------|----------|
| 任务分解（A） | 2-5s | ~10% | LLM 一次调用，无循环 |
| 检查点验证（B） | 15-20s | ~50% | VLM 批量推理 + vLLM/SGLang 加速 |
| 效率分析（C） | 1-2s | ~5% | Embedding 并行计算 |
| 轨迹差分（D） | 1-2s | ~5% | 规则引擎，确定性逻辑 |
| 综合评估（E） | <1s | ~2% | 加权聚合，纯计算 |
| **总计** | **20-30s** | **~70%** | — |
| 推理排队 | 不定 | — | 批量调度 + 异步队列 |

**吞吐保障**：
- 检查点数量自适应：粗粒度 OCR/规则预筛 → 只精判异常点
- 模块 B/C/D 并行化
- 支持离线批量处理模式

---

### KR4：输出结构化评测报告，替代人工抽检，字段完整度 ≥90%

**衡量标准**：单条报告包含以下全部字段，人工确认无需补充的比例 ≥90%。

**报告结构**：

```
📊 评估报告
├── 🎯 任务完成度（权重 50%）
│   ├── 检查点达成率：X/Y = Z%
│   ├── 核心检查点状态
│   └── 功能完备性判定：✅ 通过 / ⚠️ 部分通过 / ❌ 失败
├── ⚡ 执行效率（权重 25%）
│   ├── 总步骤数
│   ├── 重复操作率
│   ├── 无效操作列表
│   └── 循环回路标记
├── 🛡️ 异常处理（权重 15%）
│   ├── 弹窗/崩溃/权限响应检测
│   └── 异常恢复评估
├── 💡 决策合理性（权重 10%）
│   ├── 偏差类型：无影响 / 补救性 / 级联
│   └── 首错步骤 index + 根因标签
└── 📈 综合评分
    └── 加权总分 / 等级
```

**业务对接**：报告格式与 测试团队现有模板对齐，从"人工回看写结论"转为"系统自动生成 + 人工确认"。

---

### KR5：接入 测试流水线，完成 ≥50 条真实场景轨迹端到端验证

**衡量标准**：
- 在 ≥3 类真实场景（外卖、导航、设置等）中验证
- ≥50 条真实 端侧执行轨迹跑通完整判定流程
- 输出判定结果与人工评测对比的误差分析报告
- 确认系统在 现有测试流程中的可用性

## KR 汇总

| KR | 目标 | 衡量方式 | 对应模块 |
|----|------|----------|----------|
| KR1 | 覆盖 ≥6 类核心异常 | 异常类型覆盖率 | B/C/D + 达尔文VLM |
| KR2 | 准确率 ≥85%，致命召回 ≥95% | ≥200 条标注测试集 | B/D/E |
| KR3 | 单条 ≤30s，日吞吐 ≥500 | 端到端耗时 + 批量压测 | 全管线 + 推理优化 |
| KR4 | 结构化报告替代人工 | 报告字段完整度 ≥90% | E |
| KR5 | ≥50 条真实场景验证 | 线上验证通过率 | 全系统 |

## 风险与假设

| 风险/假设 | 影响 | 应对 |
|-----------|------|------|
| 达尔文 VLM 基线准确率未知 | KR2/3 目标可能偏高或偏低 | 先跑 20-30 条标注数据摸底，校准 KR 数值 |
|  轨迹格式与本系统输入不兼容 | KR5 集成受阻 | 早期对齐数据格式，定义标准轨迹 schema |
| VLM 对鸿蒙端 UI 的 Grounding 能力弱 | 检查点判定不准 | 补充鸿蒙 UI 截图做 prompt few-shot |
| 人工标注成本高（200 条） | KR2 验证延期 | 先做 50 条核心场景，后续滚动标注 |
