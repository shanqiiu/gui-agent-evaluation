# GUI Agent 评估项目当前进展与开发计划

版本：v2.0
更新日期：2026-07-20
状态：项目进展与唯一可执行计划清单；规范性架构以 `docs/01-技术方案.md` 为准，TaskGraph 优化设计以 `docs/06-GUI_Agent任务分解与规划评估优化技术方案.md` 为依据

## 1. 总体判断

当前项目已经形成一条不依赖 `src/oracle` 的新评估链，并落实了“意图匹配优先、执行证据验证在后”的两阶段 checkpoint 判定设计。

当前成熟度应定义为：功能完整且单元测试稳定的工程基线，而不是已经校准可靠的通用判定器。

主要原因：

- 当前规划表示仍是扁平 `_checkpoints`，尚未实现 TaskGraph、依赖关系、合法换序和替代分支。
- checkpoint 已完成第一轮通用化，但 RAG 知识内容仍主要覆盖 `settings`。
- 重复操作、状态边界和规划失效仍包含人工阈值或固定置信度。
- VLM 调用量约为“动作数 + 命中 checkpoint 数”，批跑成本偏高。
- 已完成批跑，但还没有形成基于人工标注集的正式指标。
- `efficiency`、`trajectory` 等模块虽已实现，但尚未接入新主链。
- 新架构回归测试为 `108 passed`。

## 2. 当前执行链路

```text
原始任务目录
  -> 预处理与动作过滤
  -> LLM+RAG 生成 checkpoints
  -> payload.json + 截图序列
  -> 图片路径转 Base64
  -> AB 相邻页面验证
  -> 视觉/OCR 状态证据
  -> checkpoint 与 agent_purpose 意图匹配
  -> checkpoint 截图执行验证
  -> 状态序列重建
  -> 重复操作判定
  -> 规划失效聚合
  -> baseline_result.json
```

主入口为 `src/evaluator/repeated_baseline.py::run_repeated_baseline`，单跑和批跑由 `start.sh` 接入。

下一代链路将在保持上述 v1 可运行的前提下增加：

```text
TaskGraph
  -> checkpoint 兼容投影
  -> 图感知对齐和 TaskProgress
  -> 多条件执行验证
  -> graph-aware planning evaluation
```

## 3. 子模块进展

| 子模块 | 状态 | 当前能力 | 主要缺口 |
|---|---|---|---|
| 启动与批跑 | 已完成 | `single/batch` 模式，自动读取 `.env` | 无并发、断点续跑、输出级别和调用缓存 |
| 原始数据预处理 | 已完成 | 解析 UTG、clearRes、OCR、actionPurpose，过滤非 GUI 节点 | 输出 `_deduped.json`、`_stategraph.json`，但新基线只消费 `payload.json` |
| 终止节点治理 | 已完成 | 过滤原始冗余 finished，只补一个终止哨兵 | 需要用真实数据继续检查提前终止和无终止场景 |
| LLM+RAG 拆解 | 已完成粒度优化 | 生成关键可观察子状态，支持空结果重试、质量校验和一次自动修正 | 输出仍为扁平 checkpoint；知识内容主要为 settings；缺少 TaskGraph 和人工指标 |
| Payload 构建 | 已完成 | 输出 `_checkpoints`、`agent_purposes`、`_ocr_pages`、源步骤索引 | 缺少 `_task_graph`、统一 schema version 和迁移机制 |
| 图片解析 | 已完成 | 路径统一转 Base64，保留原始图片引用 | 缺少图片哈希、去重缓存和损坏图片统计 |
| ABValidator | 已完成 | 对每个非终止相邻步骤调用 VLM 判断动作效果 | 几乎每步一次 VLM；页面描述存在误差传播风险 |
| 视觉证据 | 已完成 | pHash、SSIM、pixel diff、变化区域、OCR/rawPage 变化 | 权重为人工设定，尚未通过标注数据校准 |
| 子状态聚合 | 第一版完成 | 根据视觉变化、页面描述、导航动作和 checkpoint 进展划分状态 | 主链前后构建两次，视觉指标也被重复计算 |
| 意图匹配 | 已完成 | LLM 比较 checkpoint 与 agent purpose，支持连续 purpose span 和顺序约束 | 不理解依赖、合法换序和替代分支；LLM confidence 未校准 |
| 步骤对齐 | 已完成 | 将 checkpoint 对齐到真实 source step，而非均匀分配 | 一对多、多对一、隐式完成和多次尝试仍需真实数据验证 |
| CheckpointVerifier | 已完成 | 对意图召回候选使用对应前后截图和 VLM 验证 | 每个命中 checkpoint 一次 VLM；缺少多 criterion、`partial` 和缓存 |
| 重复操作判定 | 第一版完成 | 连续重复、等待重复、滑动重复、状态动作循环 | 阈值均为经验值，可靠性尚不能只凭单元测试证明 |
| 规划失效聚合 | 第一版完成 | 缺失检查点、执行阻塞、提前终止、未及时终止 | 未实际使用 TaskGraph/state sequence；部分置信度固定；首错归因不足 |
| 效率评估 | 独立实现 | 无效动作、探索开销、导航冗余、滑动效率 | 接口仍使用 `darwin_result`，未接入新基线 |
| 轨迹差分 | 独立实现 | `no_impact/remedial/cascading` 三分类 | 仍依赖 Darwin 形式的数据，未接入主入口 |
| Legacy Oracle | 隔离完成 | 只用于旧链对比和提示词参考 | 需要进一步隔离 pytest 和重复实现 |
| 输出系统 | 可用但冗余 | 每任务最多输出 9 个 JSON，批跑另有汇总 | `baseline_result.json` 已包含绝大多数子报告 |
| 文档 | 已形成分层 | `docs/01-技术方案.md` 为规范文档，优化方案提供 TaskGraph 设计 | 执行计划需要持续与实现和 benchmark 结果同步 |

## 5. 当前模型调用成本

设有效动作数为 `N`，命中 checkpoint 数为 `K`：

```text
预处理拆解：通常 1 次 LLM，空结果或质量修正时增加 1 次
意图匹配：每任务约 1 次 LLM
ABValidator：约 N 次 VLM
CheckpointVerifier：约 K 次 VLM
总模型调用：约 2 次 LLM + N + K 次 VLM
```

失败重试会进一步放大请求数。长轨迹批跑的主要成本来自 ABValidator，而不是意图匹配。

## 6. 执行原则与状态标记

- `[x]`：代码和验证已经完成。
- `[ ]`：当前可执行或满足前置条件后可执行。
- P0/P1/P2 表示交付顺序，不表示所有同级任务必须串行。
- 任何改变判定语义、证据数量或默认输出的优化，必须经过固定标注集 A/B。
- `Subtask` 是 TaskGraph 中的规划语义单元；`Checkpoint` 是现有对齐和验证链使用的兼容投影，二者不得混用。
- 未建立人工指标前，不删除 `_checkpoints`、旧 payload 支持或现有 planning subtype。
