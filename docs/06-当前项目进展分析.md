# GUI Agent 评估项目当前进展分析

版本：v1.0  
更新日期：2026-07-20  
状态：当前工程进展、成熟度与下一阶段分析

## 1. 总体结论

当前项目已经从单一规则验证脚本发展为一条相对完整、独立于旧 Oracle 的 GUI Agent 执行评估基线：

```text
原始轨迹
  -> 数据预处理
  -> LLM+RAG 任务分解
  -> 关键子状态生成
  -> 图片与页面证据解析
  -> 意图匹配
  -> 关键检查点执行验证
  -> 状态聚合
  -> 重复操作判定
  -> 规划失效判定
  -> 任务级评估报告
```

当前成熟度应定义为：

> 工程链路已经打通，核心基线可以运行，单元测试稳定，但尚未完成真实标注集上的指标校准，因此还不是生产级通用评估器。

当前 `master` 与远程仓库同步至提交 `1c31a34`。新架构测试为 `108 passed`。工作区中的未提交内容仅为 `.env` 和 `tmp` 下的本地数据文件，不属于代码进展。

## 2. 已完成能力

### 2.1 数据预处理

预处理链路已经能够完成：

- 解析任务指令、UTG、clearRes 和 OCR 信息。
- 识别实际 GUI 动作。
- 过滤 `finished`、`done`、系统消息、播报和非执行节点。
- 合并冗余的 `open_app` 节点。
- 复制截图到统一目录。
- 生成 `payload.json`、`_deduped.json` 和 `_stategraph.json`。
- 将动作、截图、页面和源步骤索引统一写入 payload。

相关实现：

- `src/preprocessor/pipeline.py`
- `src/preprocessor/preprocessor.py`
- `src/preprocessor/write_payload.py`

该部分已经具备单任务和批处理能力，并且与后续新基线的数据契约基本一致。

### 2.2 LLM+RAG 任务分解

checkpoint 粒度已经完成第一轮优化：

- checkpoint 被定义为关键、必要、可观察的任务子状态。
- 不再鼓励生成“点击按钮”“输入文本”“滑动页面”等纯动作节点。
- checkpoint 描述任务状态，不绑定固定坐标和固定操作路径。
- `expected_state` 作为后续执行验证契约。
- 增加纯操作名称、复合阶段、缺少 `expected_state`、重复状态和数量超限检查。
- 质量不合格时最多自动调用一次 LLM 修正。
- `_decomposer` 中记录质量状态、问题列表和修正状态。

相关实现：

- `src/decomposer/decomposer.py`
- `src/decomposer/knowledge_store.py`
- `src/decomposer/test_decomposer_unit.py`
- `src/preprocessor/pipeline.py`

当前限制是：代码框架已经趋于通用，但知识内容还不通用。仓库中目前只有：

```text
src/decomposer/app_knowledge/settings.md
```

因此目前只能认为 RAG 机制已经存在，尚不能认为多 App 领域知识体系已经完成。

### 2.3 图片与视觉证据

图片路径已经统一转换为 Base64，VLM 不再直接接收图片路径字符串。

当前视觉和结构证据包括：

- pHash 距离。
- SSIM。
- pixel diff。
- 页面变化区域。
- OCR/rawPage 文本变化。
- 页面状态边界置信度。
- 截图缺失和解码失败诊断。

相关实现：

- `src/common/image_resolver.py`
- `src/evaluator/visual_evidence.py`
- `src/evaluator/state_evidence.py`

这部分已经构成客观页面证据层，但视觉权重和状态边界阈值仍是经验值，尚未通过人工标注集校准。

### 2.4 意图匹配与步骤对齐

当前采用两阶段 checkpoint 判定：

```text
任务 checkpoint
  -> 与 agent_purpose / 实际状态进行意图匹配
  -> 对命中的阶段进行截图和页面执行验证
```

已具备能力：

- 使用 LLM 比较 checkpoint 与 `agent_purpose`。
- 支持一个 checkpoint 对应多个连续实际动作。
- 支持顺序约束和倒序异常。
- 支持意图未匹配状态。
- 使用真实 source step index。
- 不按动作数量平均分配 checkpoint。
- 不为未匹配 checkpoint 强行分配截图。

相关实现：

- `src/verifier/alignment.py`

该设计能够区分：

- 没有正确意图：规划或目标理解问题。
- 有正确意图但页面没有变化：执行阻塞或操作失败。
- 页面证据不足：输出不确定，而不是直接解释为正常或失败。

### 2.5 Checkpoint 执行验证

CheckpointVerifier 已经能够：

- 接收 checkpoint 的前置条件和预期状态。
- 使用对应的操作前、操作后截图。
- 调用 VLM 判断关键状态是否达成。
- 保留截图索引、动作区间和对齐信息。
- 对缺失截图返回“不确定”。
- 区分 mock 测试与真实 VLM 调用。

相关实现：

- `src/verifier/verifier.py`

当前主要问题不是功能缺失，而是 VLM 调用成本和置信度没有完成校准。

### 2.6 重复操作判定

当前重复操作基线支持：

- 连续相同动作。
- 等待或空操作重复。
- 同方向滑动重复。
- 状态与动作循环。
- 使用 checkpoint 进展抑制误报。
- 结合页面描述和状态变化判断是否真正无进展。

相关实现：

- `src/common/repeated_action_detector.py`

该模块适合作为第一版可解释规则基线，但坐标距离、文本相似度、页面相似度和重复次数仍是固定经验阈值，尚不能认为已经经过可靠性验证。

### 2.7 规划失效聚合

新规划失效模块已经接入主链，当前支持：

- 必要 checkpoint 缺失。
- 有对应意图但执行未达成。
- 提前终止。
- 完成后没有及时终止。
- 在 `baseline_result.json` 中输出规划失效结果。
- 在批跑汇总中输出规划失效标签和子类型。

相关实现：

- `src/evaluator/planning_failure.py`
- `src/evaluator/repeated_baseline.py`

该模块仍属于第一版。当前实现没有实际使用 `state_sequence` 作为规划失效证据，说明规划失效聚合尚未完全吸收状态层能力。

## 3. 尚未完成的关键部分

### 3.1 RAG 尚未形成通用知识体系

当前 RAG 存在三层缺口：

1. 知识文档数量少，目前主要是设置类知识。
2. 多 App、多版本、多语言和页面状态知识尚未建立。
3. 文档中声明了 `RAG_PERSIST_DIR`，但知识库实现仍主要使用固定目录，配置闭环尚未完成。

因此当前状态应描述为：

> RAG 框架已实现，RAG 内容、配置和持续更新体系尚未完成。

### 3.2 VLM 调用存在冗余

设有效动作数为 `N`，命中 checkpoint 数为 `K`，当前单任务模型调用量大致为：

```text
任务分解：约 1 次 LLM
意图匹配：约 1 次 LLM
ABValidator：约 N 次 VLM
CheckpointVerifier：约 K 次 VLM
```

失败重试会进一步放大请求数。长轨迹批跑的主要成本来自 ABValidator。

此外，主链会在 checkpoint 验证前后分别构建一次状态序列，导致图片解码和视觉变化指标重复计算。

当前尚未实现：

- VLM 请求缓存。
- 图片哈希与结果复用。
- AB 按需验证。
- 状态视觉证据复用。
- 批跑断点续跑。
- 模型调用数和耗时预算。

### 3.3 输出文件偏多

单个任务最多产生：

- `ab_report.json`
- `intent_matcher_diagnostics.json`
- `intent_matches.json`
- `checkpoint_alignments.json`
- `verification_report.json`
- `state_sequence.json`
- `repeated_prediction.json`
- `planning_failure_result.json`
- `baseline_result.json`

其中大部分内容已经包含在 `baseline_result.json` 中。当前输出策略更适合开发调试，不适合大规模生产批跑。

后续需要区分：

```text
compact：只保留最终结果和必要摘要
full：保留完整中间结果
debug：保留模型响应和全部诊断信息
```

### 3.4 置信度尚未完成校准

当前置信度混合了三种来源：

- VLM 自行返回的 confidence。
- 规则分数映射。
- 规划失效模块中的固定置信度常数。

这些分数目前不能直接解释为真实正确概率。需要使用人工标注集完成：

- 置信度分桶。
- Precision/Recall。
- 可靠性曲线。
- 阈值敏感性分析。
- 不同 App、任务类型和轨迹长度下的稳定性分析。

### 3.5 效率和轨迹差分未接入主链

`src/efficiency` 和 `src/trajectory` 已有实现和单元测试，但接口仍依赖 Darwin 形式的结果。

当前它们还不能直接消费新链的：

- AB 报告。
- 状态序列。
- checkpoint 验证结果。
- 新规划失效结果。

因此当前 `baseline_result.json` 还不是完整的“任务能力、合理性、效率和体验”综合报告。

- 图片路径解析。

因此目前能够证明工程实现稳定，不能证明真实场景准确率已经达到生产要求。

## 5. 下一阶段计划

### P0：建立统一评测基准

最优先需要建立人工标注集评测工具，对同一份数据运行：

- 新基线。
- 旧 Oracle 链。
- 纯规则基线。
- 不同 LLM/VLM 版本。

至少统计：

- checkpoint 召回率。
- checkpoint 达成 Precision/Recall。
- 重复操作 Precision/Recall。
- 规划失效子类型 Precision/Recall。
- 首错步骤准确率。
- 图片和 OCR 缺失率。
- VLM 不确定率。
- 单任务模型调用次数和耗时。

没有统一指标前，不应继续大规模增加异常类型或规则。

### P1：完成 RAG 闭环

- 扩展多 App 知识文档。
- 建立页面状态和功能关系知识。
- 增加版本、平台、语言和有效时间元数据。
- 完善知识库构建、发布和更新流程。
- 统计 RAG 命中率和对 checkpoint 质量的实际提升。
- 接通知识库持久化目录配置。

### P1：治理 VLM 成本

推荐实施顺序：

1. 增加图片和 VLM 请求缓存。
2. 实现 AB 按需验证模式。
3. 对意图明确且客观证据充分的阶段减少重复模型调用。
4. 复用第一次构建的视觉证据更新状态进展。
5. 批跑默认使用 compact 输出。
6. 增加断点续跑和调用预算。

### P1：校准状态和异常判定

需要重点校准：

- 视觉变化边界。
- 页面相似度阈值。
- 重复点击坐标阈值。
- 重复滑动次数。
- 任务完成率阈值。
- 规划失效置信度。
- VLM confidence 与真实准确率的映射。

### P2：接入效率和体验报告

将效率和轨迹差分模块改为直接消费新链输出，并在统一报告中增加：

```text
任务完成能力
执行合理性
规划与执行异常
执行效率
路径冗余
恢复能力
交互体验
证据完整度
```

## 6. 阶段性判断

当前项目已经完成了从原始数据预处理到重复操作和规划失效的主要工程闭环，checkpoint 粒度也已经从动作步骤转向关键可观察子状态。

当前主要优势：

- 新链独立于旧 Oracle。
- 任务意图和执行结果已经分层。
- checkpoint 不再按步骤均匀分配。
- 图片、OCR、页面变化和 VLM 证据能够统一使用。
- 单元测试覆盖较完整。
- 单跑和批跑链路已经打通。

当前主要风险：

- RAG 内容仍不够通用。
- VLM 调用量较大。
- 规则阈值没有真实标注校准。
- 效率和体验模块尚未进入最终报告。
- 规划失效尚未完全使用状态证据。

综上，项目当前最适合进入“同标注集对比评测、指标校准和成本治理”阶段，暂时不建议继续扩展更多异常类型。
