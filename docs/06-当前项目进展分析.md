# GUI Agent 评估项目当前进展分析

版本：v1.1
更新日期：2026-07-21
状态：当前工程进展、成熟度与下一阶段异常维度规划

## 1. 总体结论

当前项目已经形成一条独立于 legacy Oracle 的 GUI Agent 执行评估基线，并完成从 flat checkpoint 到 TaskGraph 双轨输出的过渡能力：

```text
原始轨迹
  -> 数据预处理
  -> LLM+RAG 任务分解
  -> TaskGraph 生成或 checkpoint 兼容迁移
  -> payload 双轨输出
  -> 图片与页面证据解析
  -> 意图匹配
  -> checkpoint 执行验证
  -> 状态聚合
  -> 重复操作判定
  -> 规划失效聚合
  -> 离线 benchmark 汇总
```

当前成熟度应定义为：工程链路已经打通，核心基线可运行，TaskGraph 契约与 benchmark 骨架已具备，但真实标注集上的指标校准、图感知进度跟踪、多维异常报告和成本治理仍未完成。

当前 `master` 已包含 P0-A/P0-B/P0-C、淘宝“再来一单”RAG 建模、planning failure 首错归因修复和 TaskGraph prompt 强化。新架构测试基线为 `137 passed`。

## 2. 已完成能力

### 2.1 数据预处理与 payload 输出

预处理链路已经能够解析任务指令、UTG、clearRes、OCR、actionPurpose 和可执行 GUI 动作，过滤非执行节点，并输出 `payload.json`、`_deduped.json`、`_stategraph.json`。

P0-B 后，payload 支持双轨输出：

- `_checkpoints`：现有 verifier 使用的兼容 checkpoint。
- `_task_graph`：`task_graph.v1` 图结构。
- `_task_graph_schema_version`：当前固定为 `task_graph.v1`。
- `_decomposer`：记录图生成状态、fallback 状态、模型配置、质量问题和修正状态。

当 `TASK_GRAPH_ENABLED=1` 时，预处理优先生成 TaskGraph；如果 LLM 输出不符合 schema，则回退到 legacy checkpoint，并迁移为 `metadata.source = "checkpoint_migration"` 的兼容 TaskGraph。

### 2.2 LLM+RAG 任务分解

Decomposer 当前有两条路径：

- `decompose()`：生成 flat checkpoint，已支持 RAG、质量校验、空结果重试和一次自动修正。
- `decompose_graph()`：生成 `task_graph.v1`，已支持 schema 校验和一次修正。

近期基于淘宝真实样本完成两项改进：

- 新增 `src/decomposer/app_knowledge/taobao.md`，把“再来一单/复购/历史订单”建模为历史订单入口、订单列表、目标订单定位、复购动作触发、订单确认页就绪。
- 强化 TaskGraph prompt，加入合法 JSON 示例，并明确禁止 `tasks/start_task/end_task/dependencies/nodes` 等非 schema 顶层字段。

当前限制是：TaskGraph 直接生成仍依赖模型遵循 schema；fallback 机制有效，但后续应减少 `checkpoint_migration`，提高 `llm_rag` 图生成成功率。

### 2.3 TaskGraph schema 与人工标注契约

已完成：

- `src/decomposer/models.py`：TaskGraph 与人工标注 dataclass。
- `src/decomposer/schema.py`：严格 JSON codec、schema version、DAG、依赖、alternative group 和人工标注校验。
- `docs/07-人工标注契约与TaskGraph-Schema.md`：远程原始轨迹到 TaskGraph/标注字段的映射，以及本地保密约束。

真实截图、UI 树、OCR 全文、服务器路径和业务敏感原始数据仍不进入本地仓库。

### 2.4 执行证据、意图匹配与 checkpoint 验证

当前主链仍采用两阶段 checkpoint 判定：

- 先用 checkpoint 与 `agent_purpose`、动作文本、页面状态做意图召回。
- 再只对召回命中的候选区间使用截图/VLM 做执行验证。

已具备能力：

- 不按动作数均分 checkpoint。
- 未匹配 checkpoint 不强行绑定截图。
- 能区分未产生正确意图、产生意图但执行未达成、证据不足。
- 支持连续 action purpose span。

当前限制是：对依赖、合法换序、替代分支、多次尝试和 partial 状态仍依赖 flat checkpoint 推断，尚未接入 TaskGraph/TaskProgress。

### 2.5 重复操作与规划失效

重复操作当前支持连续相同动作、等待重复、滑动重复、状态动作循环，并结合 checkpoint 进展抑制误报。

规划失效当前支持：

- 必要 checkpoint 缺失。
- 有意图但执行未达成，即 `execution_blocked`。
- 提前终止。
- 完成后未及时终止。

基于淘宝验证码样本，已修复主因排序：当更早的 `execution_blocked` 与后续 `missing_required_checkpoint` 同时存在时，最终 subtype 优先输出更早首错步。例如该样本从 `missing_required_checkpoint / step=16` 修正为 `execution_blocked / step=5`。

当前限制是：规划失效仍没有真正消费 TaskGraph、TaskProgress 和 state sequence 的组合证据。

### 2.6 Benchmark 骨架

已新增 `src/evaluator/benchmark.py`，支持离线读取：

- `task_annotation.json`
- `payload.json`
- `baseline_result.json`
- `planning_evaluation.json`
- `legacy_result.json`

当前最小指标包括样本数、解析失败数、schema 不兼容数、v1/v2 差异和模型调用统计。该骨架不触发 LLM/VLM 调用，适合后续接真实标注集。

## 3. 当前主要缺口

### 3.1 TaskGraph 直接生成仍不稳定

远程样本显示，模型可能输出 `{tasks,start_task,end_task}` 风格结构，导致 schema 校验失败并触发 fallback。prompt 已强化，但还需要继续在真实样本上验证：

- `graph_status=ok` 的比例。
- `graph_status=fallback_migrated` 的比例。
- 修正轮是否能把非 schema 输出转成合法 `task_graph.v1`。

### 3.2 RAG 需要从文档存在走向命中率评估

当前已有 `settings.md` 和 `taobao.md`，但还不能证明 RAG 在批量样本中稳定命中、正确引用并提升分解质量。

下一步需要统计：

- RAG 命中率。
- 错误引用率。
- `llm_rag` TaskGraph 相对 `checkpoint_migration` 的质量提升。
- App/任务类型覆盖缺口。

### 3.3 异常 taxonomy 覆盖仍不完整

后续异常建模必须以 `docs/GUI_Agent_异常Case_技术洞察.md` 1.2 “异常 case 的定义边界”为顶层 taxonomy，而不是新增一套并列分类。当前 7 类顶层异常为：

1. 循环/死循环：在有限页面集合间反复跳转，通常表现为轨迹重复、步骤数超阈值或状态循环。
2. 重复动作：对同一元素或同一语义目标重复点击、输入、等待或滑动。
3. Grounding 错误：点击坐标、输入位置、滑动方向/距离偏离目标元素。
4. 规划失效：遗漏关键步骤、依赖顺序错误、误解任务目标或进入错误分支。
5. Hallucination：幻觉出截图/OCR/UI 状态中不存在的元素、页面或能力。
6. 异常中断响应：验证码、登录、权限弹窗、Crash、网络加载等中断出现后处理异常。
7. 提前终止：任务仅部分完成即声称完成或输出 finished/done。

当前已形成第一版工程覆盖的是：重复动作、规划失效、提前终止，以及由重复检测和状态证据间接支持的循环/死循环。仍需补齐的是：Grounding 错误、Hallucination、异常中断响应，并把提前终止从 planning failure 子类型提升为统一异常事件中的顶层标签之一。

执行阻塞、恢复失败、错误级联、效率、证据完整度和高风险操作不作为 1.2 的顶层异常类别；它们应作为事件属性、二级 subtype 或报告治理维度挂载到上述 7 类之下。例如：

- 验证码、登录、权限、网络加载、App 卡死归入“异常中断响应”的 subtype；当它阻断必需 Subtask 时，也可作为“规划失效”的证据。
- 走错后返回、重试、刷新、替代路径是否有效属于 recovery outcome，用于解释首错后是否恢复，不单独作为顶层 case。
- 错误级联属于 deviation impact，用于描述首错后影响范围。
- 效率、模型调用成本、证据完整度和高风险确认属于评估报告/工程治理维度，不替代异常 taxonomy。

### 3.4 置信度与阈值尚未校准

当前置信度混合了 VLM 自评、规则分数和固定常数，不能直接解释为真实正确概率。重复操作、规划失效、视觉边界、VLM 判定都需要标注集校准。

### 3.5 成本与输出治理仍未完成

当前单任务模型调用量大致为：

```text
预处理分解：1-2 次 LLM
意图匹配：约 1 次 LLM
ABValidator：约 N 次 VLM
CheckpointVerifier：约 K 次 VLM
```

尚未完成图片哈希、VLM 缓存、AB 按需验证、状态证据复用、compact/full/debug 输出级别和断点续跑。

## 4. 下一阶段计划

### P1-A：标注与 benchmark 闭环

目标：先让异常维度可标注、可对比、可量化。

- 选取约 30 条跨 App 轨迹试标，覆盖成功、失败，以及 1.2 定义的 7 类异常：循环/死循环、重复动作、Grounding 错误、规划失效、Hallucination、异常中断响应、提前终止。
- 标注 TaskGraph、Subtask span、achieved/partial/failed/uncertain、首错步骤、恢复结果、异常顶层标签、subtype、impact 和证据引用。
- 在 benchmark 中并列读取人工标注、v1 baseline、v2 TaskGraph、legacy Oracle 和规则基线。
- 输出 Decomposer、Alignment/Verification、Planning/Recovery、Efficiency、Evidence Quality 五类指标。

### P1-B：TaskProgress 与图感知异常判定

目标：让系统从“flat checkpoint 完成率”升级为“图上任务进展”。

- 新增 `src/evaluator/task_progress.py`。
- 输出逐步骤 `TaskProgressSnapshot`：active、available、blocked、achieved、failed、uncertain Subtask。
- 支持一对多、多对一、多次尝试、合法换序、替代分支。
- planning failure 同时消费 TaskGraph、TaskProgress、state sequence、alignment 和 verification。
- 输出 `planning_evaluation.json`，定位首个违规 Subtask 和真实步骤。

### P1-C：按 1.2 七类异常补齐 taxonomy 与事件输出

目标：把异常判定从“重复操作 + 规划失效”扩展为与研究文档一致的 7 类顶层 case，并在统一事件结构中保留 subtype、恢复、级联、效率和证据质量信息。

建议实现顺序：

1. 循环/死循环：复用 state sequence、页面签名、动作循环和 checkpoint 无进展证据，输出 `loop` 事件。
2. 重复动作：保留现有 repeated detector，但把结果映射为 1.2 顶层 `repeated_action`。
3. 规划失效：接入 TaskGraph/TaskProgress，细分 missing required subtask、dependency violation、wrong branch、goal misunderstanding 和 replanning failure。
4. 提前终止：从 planning failure 子类型提升为顶层 `premature_termination` 事件，同时保留其对规划失效聚合的影响。
5. Grounding 错误：结合动作目标、点击坐标、页面变化、OCR/UI 元素变化，识别点错、输错位置、滑动方向/距离错误。
6. Hallucination：对比 agent 描述、action purpose、页面描述、OCR/UI 证据，识别不存在元素、错误页面理解和虚构能力。
7. 异常中断响应：识别验证码、登录、权限弹窗、Crash、网络加载等中断状态，以及中断后的等待、重试、放弃、绕路和错误继续。

统一事件建议写入 `planning_evaluation.json` 或新的 `anomaly_events`：

```json
{
  "category": "grounding_error",
  "subtype": "wrong_tap_target",
  "first_error_step": 5,
  "related_subtask_id": "subtask_2",
  "evidence_refs": ["state_sequence[5]", "ab_report[5]"],
  "recovery_outcome": "not_recovered",
  "impact": "cascading",
  "confidence": 0.72
}
```

`recovery_outcome`、`impact`、`efficiency_cost`、`evidence_quality` 和 `risk_level` 是事件属性或报告字段，不改变 1.2 的顶层分类。

### P1-D：RAG 与 Experience Bank

- 继续扩展多 App Markdown，但不以文档数量作为完成标准。
- 建立 generated/corrected TaskGraph 对、来源、适用 App/版本/语言、人工审核状态。
- 从首错样本提取任务模板和异常处理模板。
- 统计 RAG 命中率、错误引用率和 TaskGraph 指标净提升。

### P1-E：成本、输出与可复现性

- 增加图片内容哈希和 VLM 请求缓存。
- 复用视觉证据，避免重复构建 state sequence。
- 增加 `--output-level compact|full|debug`。
- 批跑支持断点续跑。
- 统一记录代码版本、schema 版本、模型版本、feature flag 和调用预算。

## 5. 阶段性判断

项目当前不应只继续围绕“重复操作”和“规划失效”两个标签扩展，而应按 1.2 的 7 类异常建立统一事件体系：循环/死循环、重复动作、Grounding 错误、规划失效、Hallucination、异常中断响应和提前终止。

不过实现顺序仍应克制：先用 TaskGraph/TaskProgress 建立统一进度表示，再把 recovery、cascading、efficiency、risk 和 evidence quality 作为事件属性挂到同一条证据链上。否则继续增加独立规则会造成标签互相覆盖、首错归因不稳定和报告难以校准。
